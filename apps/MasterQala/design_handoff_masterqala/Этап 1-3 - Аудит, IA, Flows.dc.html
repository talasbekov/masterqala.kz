1. Продукт в одном абзаце

MasterQala.kz — казахстанский маркетплейс бытовых услуг «на дом». Два режима: «Сейчас» — система сама находит ближайшего активного мастера волновым матчингом (3 → 6 → 10 км, таймауты 60/90 с), клиент оплачивает выезд (hold → capture), цена работ согласуется после осмотра; «Запланировать» — клиент публикует обезличенную заявку, мастера делают до 5 ставок (цена + срок + комментарий, 1 lead-кредит за отклик), клиент выбирает, мастер подтверждает за 2 ч. Бэкенд MVP полностью реализован (5 этапов: фундамент, срочный, плановый, деньги, споры). Фронтенд — 17 страниц, из которых дизайн-система применена только к 7 клиентским.

5. UX-проблемы текущего интерфейса

№ПроблемаСерьёзность
1Оффер мастеру показывает точный адрес до принятия (WorkPage рендерит offer.address) — нарушает правило защиты контакта/адреса из брифа и логику «раскрытие после принятия»Критично
2Критические статусы (отмена мастером) — через window.confirm без объяснения последствий (−2 кредита, окно блокировки)Критично
3Нет фотографий ни в срочной, ни в плановой заявке — мастер принимает решение вслепую, растёт доля споров «не та цена после осмотра»Критично
4Ставки нельзя сравнить: нет рейтинга, числа заказов, опыта, верификации, позиции цены — выбор сводится к минимальной цене (анти-цель брифа)Критично
5Нет карты нигде: ни при указании адреса, ни при поиске, ни при движении мастера (geo:update уже ходит по WS, но клиенту не показывается)Высокая
6Создание срочной заявки — одна длинная форма вместо 3–4 шагов; превью цены не объясняет структуру (выезд vs сервисный сбор vs работа после осмотра)Высокая
7Адрес — одно текстовое поле: нет подъезда/этажа/квартиры/комментария, нет сохранённых адресовВысокая
8Онбординг мастера — плоская форма; статусы NEEDS_INFO/REJECTED показаны строкой, без «что делать дальше»Высокая
9Экраны мастера, денег и вся админка — вне дизайн-системы (голый teal-700), выглядят как другой продуктВысокая
10Админка без dashboard и master-detail: оператор не видит «горящее» (поиск без мастера, просроченные споры), переходит по страницам вслепуюВысокая
11Плановая заявка: дата без слота времени, нет бюджета, нет предпросмотра перед публикацией; «срок» в ставке — свободный текстСредняя
12Нет offline/reconnecting-состояний в UI (socket connected показан только мастеру), нет push-нотификаций, нет уведомленийСредняя
13Lead-кредиты и деньги живут на разных страницах, но визуально не различены — риск смешения «кредиты ≠ тенге»Средняя

6. Противоречия UI ↔ документация

  • Спека этапа 2 описывает карточку оффера без адреса («категория, расстояние, описание, компенсация») — UI отдаёт и рендерит адрес. Чинится на уровне DTO оффера (район вместо адреса).
  • Бриф требует «две крупные карточки» режимов на главной — в UI «Запланировать» — тонкая пунктирная строка.
  • Бриф требует таймер подтверждения выбора мастером (2 ч) и таймер цены (15 мин) видимыми — бэкенд-джобы есть, но дедлайны не везде отдаются клиенту в DTO.
  • Спека споров: статус заявки при споре НЕ меняется (отдельная сущность Dispute) — фронтовые STATUS_LABELS содержат «DISPUTE» как статус заявки; в новом UI спор показываем бейджем-надстройкой, не статусом.
  • Бриф упоминает «сообщения/чат» — в системе его нет; проектируем только звонок + поддержку, чат не рисуем.
  • Рейтинг мастера — фаза 2 по спеке: в макетах закладываем слот Rating/VerificationBadge, скрываемый при отсутствии данных.

9. User flows — ключевые решения

  • Срочный заказ — 4 шага: Категория → Проблема + фото → Адрес (карта) → Расчёт и подтверждение. Расчёт всегда раскладывает: выезд N ₸ (в т.ч. сервисный сбор M ₸) + «работа — после осмотра». Отмена бесплатна до принятия — пишем это прямо на кнопке.
  • Поиск — 9 состояний (подготовка, волна 1/2/3, долго, нет мастеров, повтор, найден, едет, прибыл) — тексты волн уже есть в WAVE_TEXTS, добавляем карту и честный таймер.
  • Цена после осмотра: полноэкранная карточка с разбивкой (выезд + работа = итого), таймер 15:00, последствия отклонения («выезд не возвращается») — без dark patterns.
  • Плановый — сравнение: мобайл — карусель карточек + шторка «Сравнить (N)» с построчным сопоставлением 2–3 мастеров; desktop — таблица. Позиция цены — бейдж «ниже средней / средняя / выше средней».
  • Ставка мастера: форма всегда показывает баланс кредитов, «будет списано: 1», и подтверждение; недостаточно кредитов → инлайн-переход к покупке пакета, после покупки возврат в форму.
  • Отмены: каждая кнопка отмены показывает таблицу последствий для текущего статуса до подтверждения (двухшаговое: причина → подтвердить).
  • Спор: причина → описание → фото → отправка; вторая сторона получает один слот пояснения; оператор — timeline заказа + два независимых решения (возврат сбора / санкция) + обязательный комментарий + повторное подтверждение.
  • Верификация мастера — 6 статусов (черновик, на проверке, нужны данные, одобрен, отклонён, заблокирован); для NEEDS_INFO/REJECTED всегда: причина + конкретное следующее действие.
  • Вывод средств: сумма ≥ 5 000 ₸ → списание сразу → статус PENDING/PAID/FAILED в истории; FAILED возвращает сумму на баланс — показываем это событием в истории.

10. Обязательные состояния (матрица)

Каждый ключевой экран проектируется минимум с: loading/skeleton, empty, error (что случилось / почему / что делать), offline + reconnecting. Плюс специфичные:

ЭкранСпецифичные состояния
Поиск мастераволны 1–3, долгий поиск, нет мастеров, повторный поиск, отмена
Ожидание предложений0 ставок, новая ставка (анимация), 5/5 достигнут, срок истёк (EXPIRED)
Ставка мастеранедостаточно кредитов, лимит 5/5, повторный отклик (409), проигрыш/возврат
Оффер мастератаймер истёк, «заявку принял другой», гео запрещена, соединение потеряно
Оплата/выводprocessing, payment failed, ниже минимума, недостаточно средств, PAID/FAILED
Верификация6 статусов + upload failed + blocked (blockedUntil)
Аккаунтзаблокирован, требуется дозагрузка данных
