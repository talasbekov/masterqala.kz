# MasterQala.kz

Платформа вызова мастеров (Казахстан). Спека: `docs/project-spec.md`.

## Структура

- `apps/api` — NestJS + Prisma + PostgreSQL (API)
- `apps/web` — Vite + React PWA (клиенты, мастера, оператор)

## Запуск разработки

```bash
docker compose up -d                 # БД (5432) и тестовая БД (5433)
pnpm install
cp apps/api/.env.example apps/api/.env
cd apps/api && pnpm prisma migrate dev && pnpm prisma db seed && cd ../..
pnpm --filter api start:dev          # API на :3000
pnpm --filter web dev                # Web на :5173
```

SMS-коды в dev пишутся в лог API (`SMS → +7…`). Оператор из сидов: `+77000000001`.

## Тесты

```bash
pnpm --filter api test               # unit
pnpm --filter api test:e2e           # e2e (нужна db_test на :5433)
```

## Этап 2 — срочный режим

Полный жизненный цикл срочной заявки: создание клиентом с превью цены, волновой
матчинг мастеров, realtime-оффер (WebSocket) со звуком и таймером, принятие/отказ,
статусы «Еду» → «На месте» → цена → подтверждение клиентом → «Выполнено» →
подтверждение выполнения → закрытие, mock-оплата (HOLD/CAPTURE) и начисления
мастеру (выезд минус сервисный сбор). Отмены с обеих сторон и повторный поиск
при отсутствии мастеров — тоже часть этапа.

Что появилось:

- **Заявка и превью цены** — `POST /orders`, расчёт «Выезд: N ₸» по тарифам
  (`apps/api/src/pricing`).
- **Волновой матчинг** — очередь на pg-boss рассылает оффер ближайшим активным
  мастерам волнами, с таймаутом и переходом к следующей волне/«нет мастеров».
- **WebSocket-события** — `offer:new`, `order:status` и т.д., доставляются
  клиенту и мастеру без перезагрузки страницы (см. `apps/api/src/realtime`,
  `apps/web/src/socket`).
- **Mock-оплата и начисления** — `PaymentTransaction` (HOLD при подтверждении
  цены, CAPTURE при закрытии заявки) и `Accrual` мастеру = «выезд» минус
  сервисный сбор.
- **Экраны** — клиентские (вызов мастера, степпер статуса заявки, история),
  экран мастера («Работа»: онлайн-тумблер, полноэкранный оффер, активная
  заявка), таб-навигация («Главная», «Мои заявки», «Работа» — только для
  активных мастеров, «Профиль»).

### Как проверить руками

Нужны два окна браузера (окно A — клиент, окно B — мастер, лучше второе окно
в режиме инкогнито, чтобы не делить сессию/токен с окном A). Мастер должен
быть заранее активирован оператором (`OPERATOR_PHONE` из `.env`), геолокация
браузера разрешена в обоих окнах. Чтобы точки клиента и мастера гарантированно
совпадали, удобно подменить геолокацию через DevTools → Sensors в обоих окнах
на одну и ту же точку (например, координаты в черте Алматы).

```bash
docker compose up -d db && pnpm --filter api start:dev   # окно 1 (терминал)
pnpm --filter web dev                                     # окно 2 (терминал)
```

Чек-лист:

1. Окно B: вкладка «Работа» → «Онлайн» → индикатор соединения активен.
2. Окно A: «Вызвать мастера» → категория, описание, адрес → превью
   «Выезд: N ₸» → создать.
3. Окно B: полноэкранный оффер со звуком и таймером → «Принять».
4. Окно A: экран заявки — степпер «Принята», виден телефон мастера (пришло
   по WS, без обновления страницы).
5. Окно B: «Еду» → «На месте» → указать цену 15000 + комментарий.
6. Окно A: карточка цены с таймером → «Подтвердить».
7. Окно B: «Выполнено». Окно A: «Подтвердить выполнение» → заявка «Закрыта».
8. Проверить в БД: `PaymentTransaction` — записи HOLD и CAPTURE; `Accrual` =
   выезд минус сервисный сбор.
9. Повторить с отменами: клиент отменяет в статусе поиска (бесплатно);
   мастер отменяет после принятия (заявка возвращается в поиск и уходит
   следующему мастеру); сценарий «нет мастеров» (перевести всех мастеров в
   офлайн) → кнопка «Повторить поиск».

Примечания:

- **pg-boss** создаёт свою схему (`pgboss`) в базе автоматически при первом
  запуске API — отдельная миграция не нужна, только `docker compose up -d db`.
- В тестах/e2e очередь отключается переменной `PGBOSS_DISABLED=1` (см.
  `apps/api/.env.example`), чтобы не требовать pg-boss в CI.
- SMS-коды в dev пишутся в лог API — как и в этапе 1.
