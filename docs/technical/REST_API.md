# REST API MasterQala.kz

Документ фиксирует фактически доступные HTTP-маршруты backend-приложения на ветке `main` на момент подготовки бесплатного пилота.

## 1. Общие правила

- базовый префикс: `/api/v1`;
- формат данных: JSON, кроме загрузки и выдачи файлов;
- авторизация защищённых маршрутов: `Authorization: Bearer <accessToken>`;
- JWT выдаётся после подтверждения SMS-кода и действует 30 дней;
- глобальная валидация NestJS работает с `whitelist: true` и `transform: true`;
- поля, отсутствующие в DTO, удаляются до передачи в контроллер;
- CORS в текущей реализации разрешает любой origin;
- суммы хранятся и передаются целыми числами в тенге;
- даты передаются в ISO 8601.

## 2. Стандартные ответы об ошибках

| HTTP | Назначение |
|---|---|
| `400` | ошибка формата, DTO, номера телефона, файла или неизвестная категория |
| `401` | отсутствует, истёк или невалиден JWT |
| `403` | пользователь не является владельцем ресурса, назначенным мастером или оператором |
| `404` | заявка, файл, анкета, спор или другой ресурс не найден |
| `409` | конфликт состояния, повторное действие, проигранная гонка принятия заявки |
| `422` | бизнес-условие не выполнено: нет мастеров, нет кредитов, мастер заблокирован, достигнут лимит |
| `429` | превышен лимит запроса SMS-кода |
| `503` | внешняя финансовая операция завершилась неопределённо и требует ручной сверки |

Текст ошибки сейчас возвращается стандартным механизмом NestJS. Единого собственного envelope вида `{ code, message, details }` пока нет.

## 3. Публичные маршруты

### 3.1 Проверка состояния

| Метод | Маршрут | Ответ |
|---|---|---|
| `GET` | `/health` | `{ "status": "ok" }` |

Текущий health-check подтверждает только работу HTTP-процесса. Он не проверяет PostgreSQL, pg-boss, файловое хранилище и SMS-провайдера.

### 3.2 Авторизация

| Метод | Маршрут | Назначение |
|---|---|---|
| `POST` | `/auth/request-code` | нормализовать телефон, создать SMS-код и отправить его |
| `POST` | `/auth/verify-code` | проверить код, создать пользователя при первом входе, вернуть JWT и пользователя |

`POST /auth/request-code`:

- успешный ответ: `204 No Content`;
- срок кода: 5 минут;
- не более 3 отправок на один номер за 10 минут.

`POST /auth/verify-code`:

- успешный ответ: `200 OK`;
- не более 5 неверных попыток для найденного кода;
- ответ содержит `accessToken` и краткие данные пользователя.

### 3.3 Категории

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/categories` | получить категории услуг, отсортированные по названию |

## 4. Профиль и адреса

Все маршруты раздела требуют JWT.

### 4.1 Пользователь

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/users/me` | текущий пользователь и состояние профиля мастера |
| `PATCH` | `/users/me` | изменить разрешённые поля профиля |

Ответ `GET /users/me` содержит:

- `id`;
- `phone`;
- `name`;
- `defaultAddress`;
- `role`;
- `masterProfile.status` и `masterProfile.blockedUntil`, если анкета мастера существует.

### 4.2 Сохранённые адреса

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/addresses` | список адресов текущего пользователя |
| `POST` | `/addresses` | создать адрес |
| `PATCH` | `/addresses/:id` | изменить собственный адрес |
| `DELETE` | `/addresses/:id` | удалить собственный адрес; ответ `204` |

Адрес может включать метку, строку адреса, подъезд, этаж, квартиру, комментарий, координаты и признак адреса по умолчанию.

## 5. Анкета мастера и верификация

### 5.1 Маршруты пользователя

| Метод | Маршрут | Авторизация | Назначение |
|---|---|---|---|
| `POST` | `/masters/application` | JWT | создать или повторно отправить анкету |
| `GET` | `/masters/application` | JWT | получить собственную анкету и последний комментарий оператора |
| `POST` | `/masters/application/documents` | JWT, multipart | загрузить документ анкеты |

Документы анкеты:

- поле файла: `file`;
- разрешены JPEG, PNG и PDF;
- максимальный размер: 10 МБ;
- тип документа передаётся отдельно и соответствует Prisma enum `DocumentType`.

Документы разрешено загружать только в статусах `PENDING_REVIEW` и `NEEDS_INFO`.

### 5.2 Маршруты оператора

Все маршруты требуют JWT и роли `OPERATOR`.

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/admin/applications?status=` | список анкет, необязательный фильтр по `MasterStatus` |
| `GET` | `/admin/applications/:id` | подробности анкеты |
| `GET` | `/admin/applications/:id/documents/:docId` | открыть документ анкеты |
| `POST` | `/admin/applications/:id/decision` | одобрить, отклонить или запросить данные |

## 6. Универсальная загрузка изображений

| Метод | Маршрут | Авторизация | Назначение |
|---|---|---|---|
| `POST` | `/uploads` | JWT, multipart | сохранить JPEG или PNG и вернуть относительный `path` |

Ограничения:

- поле файла: `file`;
- разрешены JPEG и PNG;
- максимальный размер: 10 МБ.

Возвращённый путь используется при создании срочной или плановой заявки. Прямого публичного URL к upload-каталогу нет: файл выдаётся только через маршруты заявки после проверки доступа.

## 7. Срочные заявки

Все маршруты требуют JWT.

| Метод | Маршрут | Инициатор | Назначение |
|---|---|---|---|
| `POST` | `/orders/preview` | клиент | проверить наличие мастеров и получить номинальную цену выезда |
| `POST` | `/orders` | клиент | создать заявку, сохранить координаты и начать поиск |
| `GET` | `/orders/active` | клиент | получить активную срочную заявку |
| `GET` | `/orders` | клиент | история собственных срочных заявок |
| `GET` | `/orders/:id` | участник/оператор | получить заявку |
| `GET` | `/orders/:id/photos/:photoId` | участник/оператор | получить фото заявки |
| `GET` | `/master/active-order` | мастер | получить активную назначенную заявку |
| `POST` | `/orders/:id/accept` | мастер с оффером | принять заявку |
| `POST` | `/orders/:id/on-way` | назначенный мастер | `ACCEPTED → MASTER_ON_WAY` |
| `POST` | `/orders/:id/on-site` | назначенный мастер | `MASTER_ON_WAY → INSPECTION` |
| `POST` | `/orders/:id/propose-price` | назначенный мастер | предложить цену и комментарий после осмотра |
| `POST` | `/orders/:id/confirm-price` | клиент | подтвердить цену, начать работу |
| `POST` | `/orders/:id/reject-price` | клиент | отклонить цену и завершить заявку по правилам отмены |
| `POST` | `/orders/:id/complete` | назначенный мастер | отметить выполнение |
| `POST` | `/orders/:id/confirm-completion` | клиент | подтвердить выполнение и закрыть заявку |
| `POST` | `/orders/:id/cancel` | клиент/назначенный мастер | отменить в зависимости от роли и статуса |
| `POST` | `/orders/:id/retry-search` | клиент | повторить поиск после `NO_MASTERS` |

### 7.1 Ключевые ограничения

- у клиента может быть только одна активная срочная заявка;
- мастер не может принять новую срочную заявку при другой активной;
- принимать заявку может только мастер с `PENDING`-оффером текущей попытки поиска;
- гонка принятия защищена атомарным `updateMany`: проигравший получает `409`;
- цена работ подтверждается клиентом в течение 15 минут;
- `DONE` автоматически закрывается через 24 часа, если нет открытого спора;
- доступ к фото имеют только клиент заявки, назначенный мастер и оператор.

### 7.2 Поведение в `FREE_PILOT`

API и state machine сохраняются, но:

- создание заявки не должно вызывать реальный или mock `HOLD`;
- принятие не должно вызывать `CAPTURE`;
- отмена не должна вызывать `VOID`;
- закрытие не должно создавать денежное начисление мастеру;
- `calloutPrice`, `serviceFee`, `workPrice` сохраняются как аналитические значения;
- frontend должен явно показывать, что расчёт между сторонами проходит вне платформы.

## 8. Плановые заявки

Все маршруты требуют JWT.

| Метод | Маршрут | Инициатор | Назначение |
|---|---|---|---|
| `POST` | `/planned-orders` | клиент | создать и сразу опубликовать плановую заявку |
| `GET` | `/planned-orders/mine` | клиент | список собственных заявок |
| `GET` | `/planned-orders/feed` | мастер | лента опубликованных заявок по категориям мастера |
| `GET` | `/planned-orders/:id` | пользователь | получить заявку с редактированием чувствительных данных по роли |
| `GET` | `/planned-orders/:id/photos/:photoId` | участник/оператор | получить фото |
| `POST` | `/planned-orders/:id/bids` | мастер | оставить один отклик со стоимостью и сроком |
| `POST` | `/planned-orders/:id/select` | клиент | выбрать отклик |
| `POST` | `/planned-orders/:id/confirm` | выбранный мастер | подтвердить готовность |
| `POST` | `/planned-orders/:id/decline` | выбранный мастер | отказаться до подтверждения и вернуть заявку в публикацию |
| `POST` | `/planned-orders/:id/on-site` | назначенный мастер | начать выполнение |
| `POST` | `/planned-orders/:id/complete` | назначенный мастер | отметить выполнение |
| `POST` | `/planned-orders/:id/confirm-completion` | клиент | закрыть заявку |
| `POST` | `/planned-orders/:id/cancel` | клиент/назначенный мастер | отменить по доступным правилам |

### 8.1 Ключевые ограничения

- слот должен находиться в будущем и не дальше 14 дней;
- на заявку принимается максимум 5 откликов;
- один мастер может оставить только один отклик на заявку;
- выбранный мастер должен подтвердить готовность в течение 2 часов;
- опубликованная заявка без откликов становится `EXPIRED` при наступлении `slotStart`;
- `DONE` автоматически закрывается через 24 часа, если нет открытого спора;
- до назначения скрываются точный адрес, детали доступа, фото, клиент и отклики;
- клиент видит телефон мастера только начиная с `CONFIRMED`.

### 8.2 Поведение в `FREE_PILOT`

- отклик должен быть бесплатным;
- endpoint `/lead-credits/purchase` отключается;
- `placeBid` не должен уменьшать баланс;
- отсутствие `LeadCreditAccount` не должно блокировать отклик;
- лимит 5 откликов и уникальность отклика сохраняются.

## 9. Lead-кредиты

Текущая коммерческая заготовка:

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/lead-credits/balance` | баланс мастера |
| `GET` | `/lead-credits/packages` | доступные пакеты |
| `POST` | `/lead-credits/purchase` | mock-покупка пакета |

В `FREE_PILOT` рекомендуемый контракт:

- `GET /lead-credits/balance` может возвращать `null` или служебный unlimited-признак;
- `GET /lead-credits/packages` возвращает пустой список либо `featureDisabled: true`;
- `POST /lead-credits/purchase` возвращает `409 FEATURE_DISABLED_IN_FREE_PILOT`;
- frontend не отображает покупку и баланс как ограничение.

## 10. Кошелёк мастера

Текущая коммерческая заготовка:

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/wallet/balance` | баланс мастера |
| `GET` | `/wallet/withdrawals` | собственные заявки на вывод |
| `POST` | `/wallet/withdrawals` | запросить вывод |

В `FREE_PILOT`:

- денежные начисления не формируются;
- создание вывода должно быть отключено;
- frontend не должен показывать доступный к выводу баланс как реальные деньги;
- исторические записи тестовой среды нельзя переносить в пилотную production-базу как финансовые обязательства.

## 11. Споры

### 11.1 Участники заявки

| Метод | Маршрут | Назначение |
|---|---|---|
| `POST` | `/orders/:id/disputes` | открыть спор по срочной заявке |
| `POST` | `/planned-orders/:id/disputes` | открыть спор по плановой заявке |
| `POST` | `/disputes/:id/evidence` | добавить JPEG/PNG до 10 МБ |
| `PATCH` | `/disputes/:id` | вторая сторона добавляет пояснение |
| `GET` | `/disputes/:id/evidence/:docPath` | получить доказательство при наличии доступа |

Спор доступен в `IN_PROGRESS`, `DONE` и `CLOSED`. После закрытия окно открытия составляет 48 часов.

### 11.2 Оператор

Все маршруты требуют роли `OPERATOR`.

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/admin/disputes?status=` | список споров |
| `GET` | `/admin/disputes/:id` | подробности спора |
| `POST` | `/admin/disputes/:id/resolve` | разрешить спор, записать решение и санкции |

В `FREE_PILOT` флаг возврата сервисного сбора должен сохраняться только как решение оператора/аналитика либо быть недоступен, потому что платформа фактически не принимала платёж.

## 12. Отзывы

| Метод | Маршрут | Назначение |
|---|---|---|
| `POST` | `/orders/:id/review` | отзыв клиента по закрытой срочной заявке |
| `POST` | `/planned-orders/:id/review` | отзыв клиента по закрытой плановой заявке |

Одна заявка может иметь не более одного отзыва. Рейтинг используется при отображении мастеров и откликов.

## 13. Что необходимо добавить до стабильного публичного API

1. OpenAPI/Swagger, генерируемый из DTO и контроллеров.
2. Версионируемые машинные коды ошибок, не зависящие от текста.
3. Пагинацию истории, заявок оператора, споров и анкет.
4. Идемпотентный ключ для создания заявки и критичных POST-операций.
5. Ограничение CORS конкретными production-origin.
6. Расширенный health-check для БД, очереди и хранилища.
7. Явный endpoint или поле `/config/public`, сообщающее frontend режим `FREE_PILOT`.
8. Централизованный audit log операторских решений и финансовых действий.
9. Единый формат загрузки файлов и проверка фактической сигнатуры файла, а не только MIME из multipart.
10. Политика удаления аккаунта, персональных данных и файлов.
