# Стратегия тестирования MasterQala.kz

Документ определяет минимальную проверку текущей системы и первой публичной версии `FREE_PILOT`.

## 1. Текущее состояние

Backend использует:

- Jest;
- NestJS TestingModule;
- Supertest для e2e;
- отдельную PostgreSQL/PostGIS test-базу на порту `5433`;
- `PGBOSS_DISABLED=1` в тестах, где очередь не требуется.

Команды:

```bash
pnpm --filter api test
pnpm --filter api test:e2e
pnpm --filter api test:cov
```

Текущий базовый e2e smoke test проверяет только отсутствие маршрута `/`. Этого недостаточно для пилота.

Frontend сейчас не имеет зафиксированного полноценного набора unit/component/e2e-тестов в package scripts. Перед запуском требуется добавить тестирование критических пользовательских сценариев.

## 2. Цели

Тестирование должно доказать:

1. state machines не допускают некорректных переходов;
2. конкурирующие действия не создают двойное назначение или списание;
3. доступ к данным ограничен участниками и оператором;
4. realtime-события соответствуют REST-состоянию;
5. таймауты идемпотентны;
6. `FREE_PILOT` не создаёт финансовые записи;
7. платная mock-модель остаётся работоспособной для будущего включения;
8. система восстанавливается после reconnect, restart и повторной доставки jobs.

## 3. Пирамида тестов

### 3.1 Unit

Проверяют чистую бизнес-логику и небольшие сервисы:

- pricing;
- ETA;
- нормализацию телефона;
- commercial mode;
- доступность функций по режиму;
- вычисление таймаутов;
- редактирование чувствительных данных;
- правила санкций;
- правила отзывов и споров.

Unit-тесты не должны подменять проверку транзакций и уникальных ограничений.

### 3.2 Integration с PostgreSQL/PostGIS

Проверяют:

- Prisma transactions;
- raw SQL PostGIS;
- выбор кандидатов по радиусу;
- индексы и unique constraints;
- атомарные state gates;
- конкурентное принятие;
- повторную доставку jobs;
- работу миграций.

Для интеграционных тестов используется реальный PostgreSQL/PostGIS, а не SQLite.

### 3.3 API e2e

Проверяют полный HTTP-контракт через запущенное NestJS-приложение:

- авторизацию;
- роли;
- DTO validation;
- ownership;
- state transitions;
- загрузку и выдачу файлов;
- FREE_PILOT/PAID_MOCK различия.

### 3.4 WebSocket integration

Через `socket.io-client`:

- handshake;
- user rooms;
- presence;
- офферы;
- статусы;
- геолокацию;
- плановые события;
- disconnect/reconnect.

### 3.5 Frontend component

Рекомендуемый стек:

- Vitest;
- React Testing Library;
- MSW.

Проверяются:

- состояния загрузки/ошибки;
- доступность кнопок по статусу;
- FREE_PILOT-тексты;
- отсутствие коммерческих элементов;
- обработка событий;
- синхронизация после reconnect;
- защита от повторного клика.

### 3.6 Browser e2e

Рекомендуется Playwright.

Минимальные роли:

- клиент;
- активный мастер;
- второй мастер;
- оператор.

## 4. Матрица коммерческих режимов

Каждый финансово значимый сценарий тестируется минимум в двух режимах.

| Сценарий | FREE_PILOT | PAID_MOCK |
|---|---|---|
| Создание срочной заявки | нет HOLD | один успешный HOLD |
| Принятие | нет CAPTURE | один CAPTURE |
| Нет мастеров | нет VOID | один VOID |
| Бесплатная отмена | нет финансовой записи | VOID при наличии HOLD |
| Платная отмена клиента | нет Accrual | один Accrual |
| Закрытие | нет Accrual | один Accrual |
| Повторный вызов обработчика | без дублей | без дублей |
| Плановый отклик | баланс не проверяется/не меняется | списывается 1 кредит |
| Отмена клиента после выбора | финансовых движений нет | кредит возвращается |
| Покупка кредитов | запрещена | mock charge |
| Вывод | запрещён | mock payout |
| Возврат по спору | нет provider refund | mock refund |

Для `PAID_LIVE` нужны contract tests с sandbox реального провайдера, но они не входят в первую бесплатную версию.

## 5. Срочный режим: обязательные тесты

### Создание

- неизвестная категория → `400`;
- у клиента уже есть активная заявка → `409`;
- мастеров рядом нет на preview → `available: false`;
- мастеров рядом нет при create → `422`;
- координаты записаны в PostGIS;
- фото принадлежат созданной заявке;
- статус после создания — `SEARCHING`;
- отправлена wave 1 job;
- FREE_PILOT не создаёт payment transaction.

### Волны

- радиусы 3/6/10 км;
- таймауты 60/60/90 секунд;
- только `ACTIVE` и online мастера нужной категории;
- исключаются blocked мастера;
- исключаются занятые мастера;
- клиент не получает собственную заявку как мастер;
- повторная job не создаёт дубли офферов;
- конкурентные wave jobs не откатывают `wave` назад;
- после третьей безуспешной волны — `NO_MASTERS`.

### Принятие

- без оффера → `403`;
- два мастера принимают одновременно → один успех, один `409`;
- проигравшие офферы становятся `LOST`;
- проигравшие получают `offer:closed`;
- назначается правильный masterId;
- FREE_PILOT не создаёт CAPTURE.

### Выполнение

- только назначенный мастер меняет статусы;
- неверный предыдущий статус → `409`;
- клиент не может выполнить действие мастера;
- мастер не может подтвердить действие клиента;
- price timeout срабатывает один раз;
- устаревший timeout ничего не меняет;
- auto-close не работает при открытом споре;
- повторное close не создаёт дубль начисления;
- FREE_PILOT не создаёт начисление вообще.

### Отмена и повторный поиск

- клиент отменяет до принятия;
- клиент отменяет после принятия;
- мастер отменяет только в разрешённых статусах;
- мастер получает санкцию;
- заявка мастера возвращается в поиск с новой попыткой;
- отменивший мастер не получает тот же заказ повторно;
- retry доступен только клиенту в `NO_MASTERS`;
- FREE_PILOT не вызывает HOLD/VOID.

## 6. Плановый режим: обязательные тесты

### Публикация

- дата в прошлом → `400`;
- дата дальше 14 дней → `400`;
- конец раньше начала → `400`;
- заявка сразу `PUBLISHED`;
- scheduled expiry создан.

### Feed и privacy

- мастер видит только свои категории;
- точный адрес отсутствует;
- фото отсутствуют;
- данные клиента отсутствуют;
- чужие bids отсутствуют;
- бюджет и район доступны;
- неактивный/неподходящий пользователь не получает лишние данные.

### Отклики

- один отклик мастера на заявку;
- максимум пять;
- закрытая заявка не принимает ставки;
- blocked мастер не откликается;
- FREE_PILOT разрешает отклик при нулевом/отсутствующем балансе;
- PAID_MOCK атомарно списывает один кредит;
- при ошибке создания bid кредит не теряется.

### Выбор и подтверждение

- выбранный bid принадлежит заявке;
- выбранный мастер получает событие;
- остальные получают закрытие;
- confirm доступен выбранному мастеру;
- timeout 2 часа возвращает заявку в `PUBLISHED`;
- устаревший timeout после повторного выбора игнорируется;
- телефон мастера скрыт до `CONFIRMED`.

### Выполнение

- только назначенный мастер начинает и завершает;
- клиент закрывает;
- auto-close через 24 часа;
- открытый спор блокирует auto-close;
- заявка без ставок становится `EXPIRED` в slotStart.

## 7. Авторизация и безопасность

Обязательные e2e:

- неверный телефон;
- лимит SMS;
- истёкший код;
- неверный код и лимит попыток;
- повторное использование кода;
- JWT отсутствует/невалиден/истёк;
- CLIENT не вызывает operator endpoint;
- чужая заявка недоступна;
- чужой адрес нельзя изменить/удалить;
- чужой файл недоступен;
- path traversal отклоняется;
- неподдерживаемый MIME отклоняется;
- файл больше 10 МБ отклоняется;
- лишние DTO-поля не сохраняются.

## 8. Споры и отзывы

### Споры

- открыть может только участник;
- доступные статусы соблюдаются;
- после закрытия действует окно 48 часов;
- второй открытый спор блокируется;
- доказательство добавляет только участник;
- пояснение открывшей стороны блокируется;
- оператор разрешает один раз;
- auto-close заморожен при OPEN;
- FREE_PILOT не вызывает refund provider;
- решение и operatorId сохраняются.

### Отзывы

- оставить может клиент завершённой заявки;
- только один отзыв;
- рейтинг ограничен DTO;
- чужая заявка недоступна;
- рейтинг мастера корректно агрегируется между двумя типами заказов.

## 9. WebSocket

- invalid JWT → `connect_error`;
- socket входит только в собственную room;
- disconnect корректно меняет presence;
- несколько connections не должны давать ложный offline после исправления модели;
- невалидные координаты игнорируются;
- координаты вне диапазона отклоняются после добавления validation;
- `master:location` получает только клиент связанного заказа;
- события содержат документированные поля;
- REST refetch восстанавливает состояние после пропущенного события;
- повторное событие не ломает UI.

## 10. Файлы

Использовать временный каталог на каждый тестовый run.

Проверки:

- UUID-имя;
- расширение соответствует разрешённому типу;
- файл физически существует;
- DB path не абсолютный;
- удаление/rollback при ошибке БД определено;
- path traversal;
- magic bytes после реализации;
- удаление EXIF после реализации;
- backup/restore сохраняет связь записи с файлом.

## 11. Миграции

Для каждого PR со schema change:

1. новая база → все migrations применяются;
2. база предыдущей версии → upgrade проходит;
3. приложение новой версии работает после upgrade;
4. seed совместим;
5. Prisma client сгенерирован;
6. критические индексы существуют;
7. PostGIS extension и raw queries работают.

Запрещено полагаться только на `prisma db push`.

## 12. Нагрузочные сценарии пилота

До запуска выполнить хотя бы базовый load test:

- массовый вход/SMS без реальной отправки;
- 100–500 параллельных WebSocket connections;
- частые `geo:update`;
- конкурентное создание заявок;
- волновой поиск по PostGIS;
- одновременное принятие;
- плановая лента и ставки;
- загрузки файлов;
- очередь таймаутов.

Ключевые показатели:

- latency API;
- DB CPU/locks;
- количество connections;
- время wave job;
- event delivery latency;
- рост памяти;
- размер pg-boss tables.

Численные SLO утверждаются после оценки инфраструктуры пилота.

## 13. Browser e2e критического пути

### Срочный

1. мастер online;
2. клиент создаёт заказ;
3. оффер приходит со звуком/таймером;
4. мастер принимает;
5. клиент видит мастера;
6. мастер едет и прибывает;
7. предлагает цену;
8. клиент подтверждает;
9. мастер выполняет;
10. клиент закрывает и оставляет отзыв;
11. нигде нет формы платформенной оплаты;
12. в БД нет финансовых записей.

### Плановый

1. клиент публикует;
2. мастер с нулевыми кредитами видит заявку;
3. оставляет бесплатный отклик;
4. клиент выбирает;
5. мастер подтверждает;
6. адрес/телефон раскрываются в правильный момент;
7. выполнение и отзыв;
8. покупка кредитов отсутствует.

### Оператор

1. просматривает анкету;
2. открывает документ;
3. принимает решение;
4. просматривает спор;
5. разрешает спор;
6. действия появляются в audit log после его реализации.

## 14. CI gates

Каждый PR:

```text
install
lint
unit
integration/e2e
build api
build web
migration check
```

Для изменений state machine, платежей, коммерческого режима, privacy или WebSocket обязательны соответствующие e2e.

Merge блокируется при:

- failing test/build;
- незакоммиченной миграции;
- изменении endpoint/event/status без документации;
- снижении покрытия критичной ветки без объяснения;
- появлении финансовой записи в FREE_PILOT matrix.

## 15. Тестовые данные

- использовать вымышленные телефоны, ИИН, адреса и документы;
- не копировать production dump;
- оператор и мастера создаются deterministic factory/seed;
- каждый тест очищает собственные данные;
- время для timeout тестов контролируется fake timers или явным handler invocation;
- внешние SMS/payment вызовы заменяются deterministic fake/provider.

## 16. Критерии выпуска FREE_PILOT

- [ ] unit и e2e зелёные;
- [ ] оба основных browser-сценария зелёные;
- [ ] конкурентное принятие проверено;
- [ ] privacy redaction проверена;
- [ ] все FREE_PILOT финансовые инварианты проверены запросами к БД;
- [ ] purchase/withdraw backend-блокируются;
- [ ] reconnect проверен;
- [ ] migration from clean database проверена;
- [ ] backup/restore smoke test пройден;
- [ ] security P0 tests пройдены;
- [ ] известные ограничения записаны в release notes.
