# Безопасность и персональные данные MasterQala.kz

Документ фиксирует текущие защитные механизмы, известные риски и минимальный security baseline первой публичной бесплатной версии.

Это технический документ. Юридические документы, согласия и сроки хранения должны быть отдельно проверены специалистом по законодательству Казахстана.

## 1. Обрабатываемые чувствительные данные

Система хранит или передаёт:

- номер телефона пользователя;
- имя;
- ИИН и ФИО мастера;
- документы, удостоверяющие личность и квалификацию;
- домашние адреса, подъезд, этаж и квартиру;
- комментарии доступа к объекту;
- точные координаты клиента и мастера;
- описание бытовой проблемы и фотографии объекта;
- историю заказов и отмен;
- отзывы и споры;
- решения оператора;
- в коммерческих режимах — финансовые операции и выплаты.

Бесплатный пилот отключает платформенные платежи, но не уменьшает риски, связанные с идентификационными документами, адресами и геолокацией.

## 2. Текущие механизмы защиты

### 2.1 Авторизация

- вход по SMS-коду;
- нормализация номера телефона;
- код действует 5 минут;
- не более 3 отправок на номер за 10 минут;
- не более 5 попыток проверки найденного кода;
- PostgreSQL advisory lock защищает конкурентные запросы отправки кода на один номер;
- после подтверждения выдаётся JWT;
- HTTP-маршруты защищаются `JwtAuthGuard`;
- WebSocket проверяет JWT в handshake.

### 2.2 Авторизация доступа к объектам

В сервисах выполняются проверки:

- клиент является владельцем заявки;
- мастер является назначенным мастером;
- принять срочную заявку может только мастер с действующим оффером;
- операторские маршруты защищены `RolesGuard` и ролью `OPERATOR`;
- файлы заявки доступны участникам и оператору;
- доказательства спора доступны участникам и оператору;
- неназначенному мастеру плановой заявки скрываются точный адрес, детали доступа, фото, клиент и отклики;
- телефон выбранного мастера скрыт от клиента до `CONFIRMED`.

### 2.3 Защита state machine

Критичные переходы используют атомарные `updateMany` с ожидаемым предыдущим статусом. Это снижает риск:

- двойного принятия заявки;
- повторного закрытия;
- применения устаревшего таймаута;
- параллельного выбора/изменения состояния.

### 2.4 Валидация запросов

Глобальный `ValidationPipe` настроен с:

```ts
{ whitelist: true, transform: true }
```

Неописанные в DTO поля удаляются.

### 2.5 Файлы

- общая загрузка заявки: JPEG/PNG, максимум 10 МБ;
- документы мастера: JPEG/PNG/PDF, максимум 10 МБ;
- доказательства спора: JPEG/PNG, максимум 10 МБ;
- файлы сохраняются под случайным UUID-именем;
- `absolutePath()` проверяет, что итоговый путь остаётся внутри upload-каталога;
- прямой публичной раздачи upload-каталога текущим API нет.

## 3. Критические риски до публичного пилота

### P0-1. Значение JWT по умолчанию

При отсутствии `JWT_SECRET` backend использует:

```text
dev-secret-change-me
```

В production запуск с fallback должен быть запрещён. Приложение обязано завершаться с ошибкой, если секрет не задан или недостаточно устойчив.

### P0-2. CORS разрешает любой origin

HTTP API и WebSocket используют разрешение любого origin.

До публикации нужно задать allowlist, например через:

```env
CORS_ORIGINS=https://masterqala.kz,https://app.masterqala.kz
```

Неизвестный origin должен отклоняться.

### P0-3. Отсутствие production-секретов и проверки конфигурации

Нужна централизованная schema-валидация переменных окружения при старте:

- `NODE_ENV`;
- `DATABASE_URL`;
- `JWT_SECRET`;
- `COMMERCIAL_MODE`;
- `CORS_ORIGINS`;
- параметры SMS;
- `UPLOAD_DIR` или object storage;
- публичные URL frontend/backend.

### P0-4. Локальное файловое хранилище

Текущий `LocalDiskStorage` подходит для разработки и одиночного пилотного сервера, но требует:

- отдельного непубличного volume;
- резервного копирования вместе с БД;
- контроля прав файловой системы;
- запрета исполнения файлов;
- мониторинга заполнения диска;
- процедуры восстановления;
- запрета горизонтального масштабирования API без общего storage.

Для нескольких API-инстансов потребуется S3-совместимое object storage либо общий защищённый volume.

### P0-5. Проверяется MIME из multipart, а не содержимое

Пользователь может подменить заявленный MIME. Требуется:

- проверка magic bytes;
- декодирование и повторное кодирование изображений;
- антивирусная проверка PDF;
- удаление EXIF и другой метаинформации из фото;
- безопасный `Content-Disposition`;
- отдельные storage-префиксы для документов мастера, заявок и споров.

### P0-6. Нет общего rate limiting

Ограничение реализовано только для отправки SMS. Нужны лимиты как минимум для:

- verify-code;
- загрузки файлов;
- создания заявок;
- повторного поиска;
- принятия оффера;
- откликов;
- открытия споров;
- WebSocket geo:update;
- операторского входа и действий.

### P0-7. Нет подтверждённого audit log

Необходимо неизменяемо фиксировать:

- вход оператора;
- решение по анкете мастера;
- просмотр документа мастера;
- просмотр доказательства спора;
- решение по спору;
- изменение санкций;
- переключение коммерческого режима;
- будущие возвраты, выплаты и ручные корректировки.

Обычного application log для этого недостаточно.

## 4. Высокие риски

### P1-1. SMS-код хранится открытым текстом

Рекомендуется хранить HMAC/хеш кода и сравнивать безопасно. Старые коды должны регулярно удаляться.

### P1-2. JWT действует 30 дней без механизма отзыва

Текущая модель не имеет:

- refresh token;
- списка сессий;
- logout/revoke;
- версии токена пользователя;
- принудительного выхода при компрометации.

Для пилота допустим короткий access token и refresh token с ротацией либо серверная session-модель.

### P1-3. Роль в JWT может устареть

JWT содержит `role`. Если роль изменена после выдачи токена, старый token может сохранять прежние claims до истечения. Guard должен либо загружать актуального пользователя, либо использовать механизм отзыва/версии.

### P1-4. Геолокация WebSocket проверяется только на тип

Нужно проверять:

- допустимый диапазон lat/lng;
- разумную скорость перемещения;
- частоту сообщений;
- активный статус мастера;
- наличие согласия на передачу геолокации;
- прекращение обработки после завершения работы.

### P1-5. Несколько socket-соединений

Текущий disconnect переводит пользователя офлайн. Если мастер открыл приложение на двух устройствах, закрытие одного соединения может ошибочно сбросить presence. Нужен счётчик соединений или heartbeat-модель.

### P1-6. Health-check поверхностный

`GET /health` отвечает `ok`, не проверяя:

- PostgreSQL;
- PostGIS;
- pg-boss;
- доступность upload-каталога;
- свободное место;
- SMS-провайдера.

Нужны отдельные liveness и readiness probes.

### P1-7. Нет политики retention

Нужно определить сроки хранения:

- SMS-кодов;
- неотправленных/отклонённых анкет;
- документов мастера;
- координат и presence;
- фотографий заявок;
- закрытых заказов;
- споров;
- audit log;
- резервных копий.

## 5. Защита персональных данных по сценариям

### 5.1 Клиент создаёт заявку

Frontend должен объяснять:

- зачем требуется адрес и геолокация;
- когда адрес увидит мастер;
- кто увидит фотографии;
- как долго информация хранится.

Нельзя включать точный адрес в `offer:new`. Текущий оффер содержит только район, описание и расстояние — это правильная базовая модель.

### 5.2 Плановый feed

Неназначенному мастеру доступны только данные, необходимые для решения об отклике:

- категория;
- район;
- описание;
- временной слот;
- бюджет;
- количество ставок.

Точный адрес и фото должны оставаться скрытыми.

### 5.3 Документы мастера

Документы должны:

- храниться отдельно от обычных фото;
- выдаваться только оператору по авторизованному маршруту;
- логироваться при просмотре;
- иметь ограниченный срок хранения;
- не попадать в application log, error tracker и аналитические системы;
- не копироваться в тестовую среду.

### 5.4 Геолокация мастера

- используется только для presence, матчинга и активного маршрута;
- не должна публиковаться другим мастерам;
- клиент получает координаты только назначенного мастера своей заявки;
- после завершения поездки realtime-передача прекращается;
- история координат в текущей модели не хранится и не должна добавляться без отдельного решения.

## 6. Бесплатный пилот

`FREE_PILOT` должен снижать финансовую поверхность атаки:

- отсутствуют реальные реквизиты карт;
- отсутствуют HOLD/CAPTURE/VOID;
- отсутствуют выплаты;
- покупка кредитов и вывод средств блокируются backend;
- не создаются финансовые обязательства на основании mock-записей.

При этом запрещено считать режим «тестовым» с ослабленной безопасностью. В пилоте используются реальные адреса, документы и геолокация.

## 7. Минимальная production-конфигурация

```env
NODE_ENV=production
COMMERCIAL_MODE=FREE_PILOT
DATABASE_URL=postgresql://...
JWT_SECRET=<случайный секрет из secret manager>
CORS_ORIGINS=https://masterqala.kz
UPLOAD_DIR=/var/lib/masterqala/uploads
PGBOSS_DISABLED=0
```

Дополнительно должны быть настроены:

- HTTPS на reverse proxy;
- HSTS после проверки домена;
- безопасные TLS-настройки;
- firewall: публичны только 80/443;
- PostgreSQL недоступен из интернета;
- SSH по ключам, без password login;
- отдельный непривилегированный пользователь процесса;
- secret manager или защищённый env-файл вне Git;
- централизованные логи с маскированием телефона, ИИН, токенов и адресов;
- резервные копии и проверка восстановления;
- security headers;
- dependency scanning;
- регулярное обновление базового образа/ОС.

## 8. Запрещённые данные в логах

Нельзя журналировать полностью:

- JWT и Authorization header;
- SMS-код;
- ИИН;
- номер документа;
- содержимое документов;
- точный домашний адрес;
- координаты;
- фотографии;
- полные номера телефонов;
- тексты споров без редактирования.

Для корреляции использовать внутренние UUID, request ID и маскированный телефон.

## 9. Security checklist перед пилотом

### Обязательное P0

- [ ] запуск невозможен с default JWT secret;
- [ ] CORS ограничен production-origin;
- [ ] HTTPS настроен;
- [ ] БД закрыта от интернета;
- [ ] production использует отдельные секреты и базу;
- [ ] загрузки находятся в непубличном volume;
- [ ] создан backup БД и upload-каталога;
- [ ] успешно проверено восстановление;
- [ ] добавлен общий rate limit;
- [ ] проверяются сигнатуры файлов;
- [ ] операторские действия аудируются;
- [ ] `FREE_PILOT` backend-блокирует покупки и выводы;
- [ ] тестовые пользователи и mock-финансовые записи не перенесены в production.

### Рекомендуемое P1

- [ ] refresh/revoke модель токенов;
- [ ] хеширование SMS-кодов;
- [ ] readiness probes;
- [ ] object storage или формализованный single-node storage;
- [ ] Sentry/аналог с редактированием PII;
- [ ] dependency и container scanning;
- [ ] политика retention;
- [ ] процедура инцидентов и уведомления ответственных;
- [ ] ручной security review критичных маршрутов.

## 10. Реакция на инцидент

Минимальная процедура:

1. ограничить доступ или остановить затронутый сервис;
2. сохранить логи и временную шкалу без изменения исходных данных;
3. отозвать скомпрометированные секреты и токены;
4. определить затронутые аккаунты, файлы и временной период;
5. восстановить сервис из доверенного состояния;
6. оформить root cause и корректирующие действия;
7. проверить юридические обязанности по уведомлению;
8. добавить тест или автоматический контроль, предотвращающий повторение.
