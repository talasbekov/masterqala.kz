# Безопасность и персональные данные MasterQala.kz

Документ фиксирует фактические механизмы защиты, закрытые риски и обязательный security baseline первой публичной версии MasterQala.kz.

Это технический документ. Юридические основания обработки, тексты согласий, трансграничная передача и сроки хранения должны отдельно проверяться специалистом по законодательству Республики Казахстан.

## 1. Модель угроз пилота

Даже без встроенных платежей система обрабатывает данные высокой чувствительности:

- номер телефона и имя;
- ИИН и ФИО мастера;
- документы, удостоверяющие личность и квалификацию;
- домашние адреса, подъезд, этаж и квартиру;
- комментарии для доступа к объекту;
- координаты клиента и мастера;
- фотографии жилых и коммерческих помещений;
- описание бытовой проблемы;
- историю заявок, отмен и санкций;
- отзывы и споры;
- решения операторов;
- в `PAID_MOCK` и будущем `PAID_LIVE` — финансовые операции и выплаты.

`FREE_PILOT` уменьшает финансовую поверхность атаки, но не делает систему тестовой или нечувствительной: реальные документы, адреса и геолокация требуют production-уровня защиты.

## 3. Текущие механизмы защиты

### 3.1 SMS-аутентификация

Реализовано:

- нормализация номера телефона;
- шестизначный код;
- срок действия 5 минут;
- максимум 3 отправки на один номер за 10 минут;
- максимум 5 попыток проверки актуального кода;
- PostgreSQL advisory lock для конкурентных запросов отправки;
- одноразовое использование записи кода.

Ограничения:

- код хранится открытым текстом;
- отдельный rate limit на `verify-code` отсутствует;
- нет device/session fingerprint;
- старые записи не очищаются автоматически.

### 3.2 HTTP JWT

После успешного SMS-входа выдаётся JWT на 30 дней.

`JwtAuthGuard`:

1. проверяет Bearer token;
2. валидирует подпись JWT;
3. извлекает `sub`;
4. загружает актуального пользователя из PostgreSQL;
5. помещает текущую запись пользователя в request.

Это означает, что HTTP-авторизация не доверяет устаревшей роли из token payload: `RolesGuard` работает с актуальным пользователем из БД.

Остаются риски:

- нет refresh token;
- нет logout/revoke;
- нет списка сессий;
- нет принудительного отзыва токена;
- компрометированный token действует до 30 дней.

### 3.3 Socket.IO

Handshake проверяет JWT и добавляет соединение в комнату:

```text
user:{userId}
```

Пользовательские комнаты позволяют отправлять события только адресату.

Ограничения:

- WebSocket не использует полноценную session/revoke модель;
- presence-команды требуют дополнительных rate limits;
- несколько одновременных socket-соединений одного пользователя не учитываются счётчиком;
- disconnect одного устройства может ошибочно перевести мастера офлайн;
- необходимо повторно проверять рабочий статус мастера при чувствительных realtime-действиях.

### 3.4 Объектная авторизация

Сервисы проверяют:

- клиент владеет заявкой;
- мастер назначен на заявку;
- срочную заявку принимает только мастер с актуальным `PENDING`-оффером;
- операторские endpoints требуют `OPERATOR`;
- фото заявки доступны участникам и оператору;
- доказательства спора доступны участникам и оператору;
- точный адрес и фото плановой заявки скрыты до назначения;
- телефон выбранного мастера скрыт от клиента до `CONFIRMED`.

Object-level проверки должны оставаться в backend. Скрытие кнопки или поля на frontend не является контролем доступа.

### 3.5 State machine и гонки

Критичные переходы используют транзакции и атомарные `updateMany` с ожидаемым предыдущим статусом.

Это защищает от:

- двойного принятия заявки;
- повторного закрытия;
- устаревшей queue-job;
- конкурентного выбора мастера;
- повторного списания/возврата в штатных сервисных ветках.

DB-level idempotency финансовых операций всё ещё требует усиления до `PAID_LIVE`.

### 3.6 Валидация DTO

Глобальный `ValidationPipe`:

```ts
{ whitelist: true, transform: true }
```

Неизвестные поля удаляются. DTO ограничивают длину строк, UUID, числовые диапазоны, количество фото и enum-значения.

Необходимо дополнительно:

- включить `forbidNonWhitelisted` для внешнего API либо формализовать текущее тихое удаление;
- нормализовать/очищать пользовательский текст перед дальнейшим отображением;
- добавить единые машинные коды ошибок;
- ограничить размеры JSON body на reverse proxy и в NestJS.

## 4. Минимизация данных в realtime

### 4.1 Срочный оффер мастеру

`offer:new` до принятия содержит только данные, необходимые для решения:

- `orderId`;
- категорию;
- описание;
- район;
- примерное расстояние;
- дедлайн;
- волну;
- коммерческие capability-поля.

Запрещено передавать до принятия:

- точный адрес;
- подъезд;
- этаж;
- квартиру;
- комментарий доступа;
- координаты клиента;
- телефон клиента;
- исходные фотографии с геометаданными.

В PR #4 добавлен регрессионный тест отсутствия этих полей в `offer:new`.

### 4.2 Плановая лента

Неназначенному мастеру доступны только:

- категория;
- район;
- описание;
- временной слот;
- бюджет;
- количество ставок;
- `commercialMode` для определения стоимости отклика.

Точный адрес, детали доступа, фото, клиент и чужие отклики должны оставаться скрытыми.

### 4.3 Геолокация мастера

Координаты используются для:

- presence;
- PostGIS-матчинга;
- передачи позиции назначенного мастера клиенту активной заявки;
- расчёта ETA.

Координаты не должны:

- передаваться другим мастерам;
- сохраняться как долгосрочная история без отдельного решения;
- поступать после завершения/отмены активного маршрута;
- попадать в application logs и error tracker.

## 5. Коммерческий режим и безопасность

### 5.1 `FREE_PILOT`

Реализованные гарантии:

- отсутствуют HOLD/CAPTURE/VOID/refund;
- не создаются `PaymentTransaction`;
- не создаются `Accrual`;
- не увеличивается кошелёк мастера;
- не создаются покупки lead-кредитов;
- плановый отклик не списывает и не возвращает кредит;
- вывод средств блокируется;
- HTTP и Socket.IO показывают нулевые суммы выезда/сбора;
- режим хранится в заявке и не меняется после переключения env.

### 5.2 `PAID_MOCK`

Mock-записи нельзя:

- считать реальным подтверждением платежа;
- переносить в production как финансовое обязательство;
- смешивать с аналитикой пилота без фильтра `commercialMode`;
- показывать пользователю как реальные деньги в `FREE_PILOT`.

### 5.3 `PAID_LIVE`

Enum существует для миграционной стабильности, но API прекращает запуск, пока реальный адаптер не подключён.

До включения обязательны:

- PCI/провайдерская модель без хранения карточных данных в MasterQala;
- provider webhooks с проверкой подписи;
- idempotency keys;
- ledger и сверка;
- обработка неопределённых исходов;
- возвраты и dispute reconciliation;
- отдельный threat model.

## 6. Файлы

### 6.1 Текущее поведение

- изображения заявки: JPEG/PNG, до 10 МБ;
- документы мастера: JPEG/PNG/PDF, до 10 МБ;
- доказательства спора: JPEG/PNG, до 10 МБ;
- случайное имя файла;
- относительный путь хранится в БД;
- `absolutePath()` должен удерживать путь внутри upload-каталога;
- upload-каталог не раздаётся как публичная static-директория;
- выдача выполняется через авторизованные endpoints.

### 6.2 Незакрытые P0

Сейчас доверие строится на MIME из multipart. Перед публичным запуском нужны:

- проверка magic bytes;
- безопасное декодирование и повторное кодирование изображений;
- удаление EXIF/GPS metadata;
- проверка PDF антивирусом/санитайзером;
- запрет SVG, HTML и исполняемых форматов;
- безопасный `Content-Disposition`;
- отдельные storage-prefix/ACL для заявок, споров и документов мастеров;
- checksum;
- quota пользователя и общий disk quota;
- удаление orphan-файлов при rollback/ошибке транзакции.

### 6.3 Local disk

Для single-node пилота допустим только при условиях:

- отдельный непубличный volume;
- права только для пользователя API;
- `noexec`/эквивалентная защита;
- резервное копирование вместе с БД;
- проверенное восстановление;
- мониторинг свободного места;
- запрет запуска второго API-инстанса без общего storage.

Для горизонтального масштабирования требуется S3-совместимое object storage или общий защищённый volume.

## 7. Критические P0 до публичного production

### P0-1. Запрет default JWT secret

Сейчас при отсутствии секрета используется development fallback.

Production должен завершать запуск, если:

- `JWT_SECRET` отсутствует;
- секрет слишком короткий/предсказуемый;
- используется известное development-значение.

### P0-2. CORS allowlist

HTTP и Socket.IO сейчас разрешают любой origin.

Нужно:

```env
CORS_ORIGINS=https://masterqala.kz,https://app.masterqala.kz
```

Неизвестный origin должен отклоняться. Для mobile/native-клиента правила следует определить отдельно, не открывая browser CORS всем доменам.

### P0-3. Валидация окружения

При старте должны проверяться:

- `NODE_ENV`;
- `DATABASE_URL`;
- `JWT_SECRET`;
- `COMMERCIAL_MODE`;
- `CORS_ORIGINS`;
- SMS credentials/provider mode;
- `UPLOAD_DIR` или object storage;
- frontend/backend public URLs;
- queue configuration.

`COMMERCIAL_MODE` уже валидируется, включая fail-fast `PAID_LIVE`; остальная конфигурация пока не объединена в одну schema.

### P0-4. Общий rate limiting

Нужны лимиты для:

- `request-code` и `verify-code`;
- uploads;
- создания заявок;
- retry-search;
- принятия оффера;
- плановых откликов;
- споров и доказательств;
- `presence:online` и `geo:update`;
- операторских endpoints.

Rate limit должен учитывать IP, userId, телефон и endpoint в зависимости от сценария.

### P0-5. Security audit log

Неизменяемо фиксировать:

- вход и чувствительные действия оператора;
- просмотр документа мастера;
- решение по анкете;
- просмотр доказательства;
- решение по спору;
- изменение санкций;
- изменение production-конфигурации/режима;
- будущие возвраты и выплаты;
- административное изменение данных заявки.

Обычный stdout/application log недостаточен.

### P0-6. Backup и восстановление

Обязательны:

- резервные копии PostgreSQL;
- резервные копии файлового volume;
- шифрование backup;
- ограничение доступа;
- retention;
- регулярная фактическая проверка restore;
- документированный RPO/RTO.

Backup, который не проходил restore-test, не считается готовым.

## 8. Высокие риски P1

### P1-1. SMS-коды

- хранить HMAC/хеш вместо plaintext;
- очищать истёкшие записи;
- не включать код в логи;
- отслеживать аномалии verify-attempts.

### P1-2. Сессии

- короткий access token;
- refresh token с ротацией;
- session table;
- logout/revoke;
- принудительный отзыв всех сессий;
- device metadata без избыточного fingerprinting.

### P1-3. WebSocket presence

- heartbeat;
- счётчик соединений пользователя;
- диапазон lat/lng;
- rate limit;
- проверка невозможной скорости;
- актуальный `MasterProfile.status`;
- остановка relay после завершения заявки.

### P1-4. Readiness

Отдельные endpoints:

- liveness — процесс отвечает;
- readiness — доступны PostgreSQL/PostGIS, queue и writable storage;
- dependency status не должен раскрывать секреты наружу.

### P1-5. Retention

Определить сроки хранения:

- SMS-кодов;
- отклонённых/неполных анкет;
- документов мастера;
- адресов и фото;
- presence;
- заказов;
- споров;
- audit log;
- финансовых журналов;
- резервных копий.

Удаление должно распространяться на БД, object/local storage, backup по утверждённой политике и аналитические копии.

## 9. Запрещённые данные в логах

Нельзя журналировать полностью:

- JWT;
- Authorization/Cookie headers;
- SMS-код;
- ИИН;
- реквизиты документов;
- содержимое документов;
- точный адрес;
- координаты;
- фотографии;
- полный телефон;
- свободный текст спора без редактирования;
- provider secrets и webhook signatures.

Для корреляции использовать:

- request ID;
- внутренние UUID;
- маскированный телефон;
- тип события;
- безопасный machine error code после его внедрения.

## 10. Минимальная production-конфигурация

```env
NODE_ENV=production
COMMERCIAL_MODE=FREE_PILOT
DATABASE_URL=postgresql://...
JWT_SECRET=<случайный секрет из secret manager>
CORS_ORIGINS=https://masterqala.kz
UPLOAD_DIR=/var/lib/masterqala/uploads
PGBOSS_DISABLED=0
```

Инфраструктура:

- HTTPS reverse proxy;
- HSTS после проверки домена;
- актуальный TLS;
- наружу открыты только 80/443;
- PostgreSQL и queue недоступны из интернета;
- SSH только по ключам;
- непривилегированный пользователь процесса;
- секреты вне Git;
- security headers;
- dependency/container scanning;
- обновление ОС и образов;
- централизованные логи с PII-redaction;
- backup и restore-test.

## 11. Security gate перед запуском

### Обязательные P0

- [ ] запуск невозможен с default/слабым JWT secret;
- [ ] CORS ограничен production-origin;
- [ ] env schema валидирует обязательные настройки;
- [ ] HTTPS и security headers настроены;
- [ ] PostgreSQL закрыт от интернета;
- [ ] upload volume непубличен и защищён;
- [ ] magic bytes/EXIF/PDF проверки реализованы;
- [ ] общий rate limit включён;
- [ ] операторские действия аудируются;
- [ ] backup БД и файлов создан;
- [ ] restore фактически проверен;
- [x] `FREE_PILOT` блокирует покупки, выплаты и финансовые side effects;
- [x] HTTP и Socket.IO не показывают номинальный сбор как оплату;
- [x] точный адрес отсутствует в `offer:new` до принятия;
- [ ] тестовые пользователи, документы и mock-финансы не перенесены в production;
- [ ] ручной security review проведён перед открытием доступа.

### Рекомендуемые P1

- [ ] session/refresh/revoke;
- [ ] хеширование SMS-кодов;
- [ ] liveness/readiness;
- [ ] heartbeat и multi-socket presence;
- [ ] object storage либо формализованный single-node storage;
- [ ] error tracking с PII-redaction;
- [ ] dependency/container scanning;
- [ ] retention и account deletion process;
- [ ] incident response drill.

## 12. Реакция на инцидент

Минимальный порядок:

1. ограничить доступ или остановить затронутый компонент;
2. сохранить временную шкалу и доказательства без изменения исходных данных;
3. отозвать секреты, токены и доступы;
4. определить затронутые аккаунты, файлы, заявки и период;
5. оценить распространение данных в logs, backups и аналитике;
6. восстановить сервис из доверенного состояния;
7. выполнить юридическую оценку обязанностей по уведомлению;
8. оформить root cause и корректирующие действия;
9. добавить автоматический контроль или тест против повторения;
10. проверить закрытие действий независимым reviewer.
