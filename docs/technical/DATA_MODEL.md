# Модель данных MasterQala.kz

Документ описывает фактическую Prisma-модель и её назначение. Источник истины — `apps/api/prisma/schema.prisma`.

## 1. Общие соглашения

- СУБД: PostgreSQL.
- Геоданные: PostGIS `geography(Point, 4326)`.
- Первичные ключи бизнес-сущностей: UUID в строковом представлении.
- Денежные значения: `Int`, единица — тенге.
- Время: `DateTime`, хранится в UTC; локализация выполняется на frontend.
- Связи с файлами хранят относительный путь, бинарные данные находятся вне PostgreSQL.
- Prisma не управляет PostGIS-полями напрямую: они объявлены как `Unsupported` и изменяются raw SQL-запросами.

## 2. Пользователи и авторизация

### `User`

Единая учётная запись клиента, мастера и оператора.

Ключевые поля:

- `phone` — уникальный номер;
- `name`;
- `defaultAddress` — устаревающее строковое поле, параллельно существует нормализованная таблица `Address`;
- `role` — `CLIENT` или `OPERATOR`.

Мастер не является отдельной ролью пользователя. Возможность работать мастером определяется наличием `MasterProfile` со статусом `ACTIVE`.

### `SmsCode`

Одноразовые коды входа:

- номер телефона;
- шестизначный код;
- срок действия;
- число неуспешных попыток;
- время использования.

Индекс: `(phone, createdAt)`.

Риски текущей модели:

- код хранится открытым текстом;
- нет отдельного поля назначения кода;
- нет автоматической очистки старых записей;
- нет device/session модели и refresh token.

## 3. Категории и мастера

### `Category`

Справочник услуг:

- уникальный `slug`;
- отображаемое `name`.

Связан со срочными заявками, плановыми заявками и компетенциями мастеров.

### `MasterProfile`

Анкета и рабочее состояние мастера:

- `userId` — уникальная связь с `User`;
- ФИО, ИИН, район, стаж;
- статус проверки;
- причина отклонения;
- `priorityPenaltyUntil`;
- `blockedUntil`.

Статусы:

- `PENDING_REVIEW`;
- `NEEDS_INFO`;
- `ACTIVE`;
- `REJECTED`.

### `MasterCategory`

Связь many-to-many мастера с категориями.

Составной первичный ключ:

```text
(masterProfileId, categoryId)
```

### `MasterDocument`

Метаданные документов анкеты:

- тип `ID_CARD` или `QUALIFICATION`;
- относительный путь;
- исходное имя;
- MIME;
- размер.

### `VerificationDecision`

Неизменяемая история решений оператора:

- анкета;
- оператор;
- решение `APPROVE`, `REJECT` или `REQUEST_INFO`;
- комментарий;
- время.

## 4. Срочные заявки

### `Order`

Главная сущность срочного режима.

Группы полей:

**Участники**

- `clientId`;
- `masterId` — nullable до принятия.

**Задача и адрес**

- категория;
- описание;
- строка адреса;
- район;
- подъезд, этаж, квартира, комментарий;
- `location` PostGIS.

**Поиск**

- `status`;
- `wave`;
- `searchAttempt`.

**Экономика**

- `calloutPrice`;
- `serviceFee`;
- `workPrice`;
- `workComment`.

**Аудит жизненного цикла**

- причина отмены;
- `acceptedAt`;
- `onSiteAt`;
- `priceProposedAt`;
- `completedAt`;
- `closedAt`;
- `createdAt`, `updatedAt`.

Индексы:

- `(clientId, status)`;
- `(masterId, status)`.

Статусы:

```text
CREATED
SEARCHING
ACCEPTED
MASTER_ON_WAY
INSPECTION
AWAITING_PRICE_CONFIRM
IN_PROGRESS
DONE
CLOSED
NO_MASTERS
CANCELLED_BY_CLIENT
CANCELLED_BY_MASTER
DISPUTE
```

Фактическое замечание: текущая отмена мастером возвращает заявку в `SEARCHING`, поэтому `CANCELLED_BY_MASTER` пока не используется как конечный переход.

### `OrderOffer`

Факт отправки оффера конкретному мастеру:

- заявка;
- мастер;
- волна;
- попытка поиска;
- время отправки и ответа;
- результат `PENDING`, `ACCEPTED`, `LOST`, `EXPIRED`.

Критическое ограничение идемпотентности:

```text
unique(orderId, masterUserId, attempt)
```

Оно не позволяет повторно создать оффер одному мастеру в рамках одной попытки поиска.

### `MasterPresence`

Текущее presence-состояние мастера:

- первичный ключ `masterUserId`;
- онлайн/офлайн;
- время последней активности;
- PostGIS-координата.

История перемещений не хранится.

### `OrderPhoto`

Относительные пути фото срочной заявки.

## 5. Плановые заявки

### `PlannedOrder`

Основные поля:

- клиент, категория;
- описание и адрес;
- предполагаемый бюджет;
- начало и конец временного слота;
- статус;
- назначенный мастер;
- выбранный отклик;
- согласованная цена;
- причина отмены;
- timestamps жизненного цикла.

Статусы:

```text
CREATED
PUBLISHED
MASTER_SELECTED
CONFIRMED
IN_PROGRESS
DONE
CLOSED
EXPIRED
CANCELLED_BY_CLIENT
CANCELLED_BY_MASTER
DISPUTE
```

Фактическое замечание: сервис создаёт запись сразу в `PUBLISHED`, поэтому `CREATED` сейчас не используется как отдельный пользовательский этап.

Индексы:

- `(clientId, status)`;
- `(masterId, status)`;
- `(categoryId, status)`.

### `PlannedOrderBid`

Отклик мастера:

- заявка;
- мастер;
- цена;
- срок выполнения в текстовом виде;
- комментарий;
- время создания.

Ограничение:

```text
unique(plannedOrderId, masterUserId)
```

Один мастер не может оставить два отклика на одну заявку.

`selectedBidId` в `PlannedOrder` уникален и указывает на выбранный отклик.

### `PlannedOrderPhoto`

Относительные пути фото плановой заявки.

## 6. Адреса

### `Address`

Сохранённые адреса пользователя:

- пользователь;
- метка;
- адрес;
- детали доступа;
- координаты `Float`;
- признак по умолчанию.

Текущая модель не гарантирует на уровне БД, что у пользователя только один `isDefault = true`. Это обеспечивается сервисной логикой и должно покрываться транзакционными тестами.

## 7. Платежи и начисления срочного режима

### `PaymentTransaction`

Журнал mock/будущих платёжных операций:

- заявка;
- тип `HOLD`, `CAPTURE`, `VOID`;
- сумма;
- статус `PENDING`, `SUCCEEDED`, `FAILED`;
- `providerRef`;
- время.

Индекс: `(orderId, type)`.

Уникального ограничения на тип операции нет. Идемпотентность сейчас обеспечивается предварительным поиском существующей операции в provider-сервисе, а не БД.

### `Accrual`

Номинальное начисление компенсации за выезд мастеру.

Критическое ограничение:

```text
orderId @unique
```

Оно обеспечивает не более одного начисления на срочную заявку.

Тип начисления сейчас один: `CALLOUT_COMPENSATION`.

## 8. Lead-кредиты планового режима

### `LeadCreditAccount`

Текущий баланс мастера. Первичный ключ — `masterUserId`.

### `LeadCreditTransaction`

История движения кредитов:

- `PURCHASE`;
- `SPEND`;
- `REFUND`;
- `PENALTY`.

Может ссылаться на отклик или покупку только идентификатором без Prisma relation.

### `LeadCreditPurchase`

Покупка пакета:

- число кредитов;
- цена;
- статус оплаты;
- ссылка провайдера.

## 9. Кошелёк и выводы

### `MasterWalletAccount`

Доступный баланс мастера. Первичный ключ — `masterUserId`.

### `WithdrawalRequest`

Запрос вывода:

- сумма;
- статус `PENDING`, `PAID`, `FAILED`;
- reference провайдера;
- время запроса и выплаты.

## 10. Споры

### `Dispute`

Спор относится ровно к одной из сущностей:

- `orderId`;
- либо `plannedOrderId`.

Поля:

- открывший пользователь;
- роль открывшего `CLIENT` или `MASTER`;
- причина;
- массив путей доказательств;
- пояснение второй стороны;
- статус `OPEN` или `RESOLVED`;
- решение о возврате сервисного сбора;
- решение о санкции мастеру;
- текст решения;
- оператор и время разрешения.

На уровне Prisma нет check constraint, гарантирующего, что заполнено ровно одно из `orderId`/`plannedOrderId`. Инвариант обеспечивает сервис.

Текущий код рассчитывает на обработку Prisma `P2002` при повторном открытии спора, но в показанной Prisma-модели нет уникального ограничения на `orderId` или `plannedOrderId`. Перед пилотом необходимо проверить фактическую миграцию и добавить частичные уникальные индексы для одного открытого спора на заявку.

## 11. Отзывы

### `Review`

- относится к срочной или плановой заявке;
- `orderId` и `plannedOrderId` по отдельности уникальны;
- клиент-автор;
- мастер-получатель;
- рейтинг;
- комментарий.

Как и у спора, модель допускает теоретическое заполнение обоих nullable target-полей. Сервис должен сохранять ровно один target.

## 12. Санкции мастера

### `MasterCancellation`

Аудит отмен мастером:

- мастер;
- тип заявки `URGENT` или `PLANNED`;
- идентификатор заявки строкой;
- время.

Прямая foreign key связь с `Order`/`PlannedOrder` отсутствует, поскольку target полиморфный.

## 13. Поведение модели в `FREE_PILOT`

### 13.1 Сохраняются без изменений

- пользователи и SMS-вход;
- анкеты мастеров, документы и решения;
- категории;
- срочные и плановые заявки;
- офферы и отклики;
- presence;
- адреса и фото;
- споры;
- отзывы;
- отмены и санкции;
- номинальные `calloutPrice`, `serviceFee`, `workPrice`, бюджет и цена отклика.

### 13.2 Не должны создаваться как реальные финансовые обязательства

- `PaymentTransaction`;
- `Accrual`;
- `LeadCreditPurchase`;
- `LeadCreditTransaction` типа `SPEND`/`PURCHASE`;
- новые `WithdrawalRequest`;
- увеличение реального `MasterWalletAccount.balance`.

### 13.3 Рекомендуемая реализация

Не удалять финансовые таблицы. Они нужны для будущего коммерческого режима и миграции без смены модели.

Добавить явный режим приложения:

```text
COMMERCIAL_MODE = FREE_PILOT | MOCK_COMMERCIAL | LIVE_COMMERCIAL
```

Финансовые сервисы должны использовать стратегию по режиму, а не условные проверки, разбросанные по `OrdersService`, `PlannedOrdersService`, `WalletService` и `DisputesService`.

## 14. Инварианты, обязательные для тестов

1. Один активный срочный заказ клиента.
2. Один активный срочный заказ назначенного мастера.
3. Один оффер мастеру на попытку поиска.
4. Один отклик мастера на плановую заявку.
5. Выбранный отклик принадлежит выбираемой заявке.
6. Только участники и оператор получают файлы заявки и спора.
7. Один отзыв на заявку.
8. Не более одного начисления за срочную заявку.
9. Статус меняется только из разрешённого предыдущего состояния.
10. Финансовые таблицы не изменяются в `FREE_PILOT`.
11. Закрытие через таймаут не выполняется при открытом споре.
12. Точный адрес плановой заявки скрыт от неназначенного мастера.
13. Один открытый спор на одну заявку — после добавления DB-ограничения.
14. У пользователя не более одного адреса по умолчанию — после усиления DB-инварианта.

## 15. Рекомендуемые изменения схемы до production

- добавить миграционно безопасные ограничения для споров;
- добавить constraint «ровно один target» для `Dispute` и `Review`;
- определить unique/idempotency constraints для платёжных операций;
- добавить audit log критичных изменений;
- вынести историю изменения статусов заявок в отдельную таблицу;
- добавить soft-delete/retention поля для персональных данных;
- предусмотреть object storage metadata и checksum файлов;
- определить версию тарифов, по которой рассчитаны `calloutPrice` и `serviceFee`;
- хранить структурированный срок планового отклика вместо произвольного `term`;
- добавить optimistic version либо event sequence для синхронизации REST и WebSocket.
