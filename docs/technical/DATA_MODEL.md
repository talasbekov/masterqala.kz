# Модель данных MasterQala.kz

Документ описывает фактическую Prisma/PostgreSQL-модель MasterQala.kz после реализации бесплатного пилота в PR #4.

Источник истины:

1. `apps/api/prisma/schema.prisma`;
2. миграции `apps/api/prisma/migrations/**`;
3. сервисы, которые создают и изменяют записи.

До объединения PR #4 ветка `main` может не содержать `CommercialMode` и связанные поля.

## 1. Общие соглашения

- СУБД: PostgreSQL 16;
- геоданные: PostGIS `geography(Point, 4326)`;
- идентификаторы бизнес-сущностей: UUID в строковом представлении;
- денежные значения: `Int`, единица — тенге;
- время: Prisma `DateTime`, хранение в UTC;
- пользовательская локализация времени выполняется на клиенте;
- файлы не хранятся в PostgreSQL: в таблицах сохраняются относительные пути и метаданные;
- PostGIS-поля объявлены через `Unsupported(...)` и изменяются raw SQL-запросами;
- критические переходы статусов выполняются атомарными `updateMany`/транзакциями.

## 2. Коммерческий режим

### 2.1 Enum `CommercialMode`

```text
FREE_PILOT
PAID_MOCK
PAID_LIVE
```

Назначение:

- `FREE_PILOT` — платформа не принимает и не выплачивает деньги, не списывает lead-кредиты;
- `PAID_MOCK` — тестовый коммерческий контур с mock-провайдером и финансовыми записями;
- `PAID_LIVE` — зарезервирован для реального провайдера. Текущий API намеренно прекращает запуск при выборе этого режима.

### 2.2 Режим хранится в заявке

Поля:

```text
Order.commercialMode
PlannedOrder.commercialMode
```

Оба поля:

- обязательные;
- имеют Prisma default `PAID_MOCK`;
- индексируются вместе с `createdAt`;
- автоматически получают текущий режим приложения при создании записи;
- не должны изменяться после создания заявки.

Индексы:

```text
Order(commercialMode, createdAt)
PlannedOrder(commercialMode, createdAt)
```

Это позволяет:

- разделять пилотную и коммерческую аналитику;
- безопасно переключать режим для новых заявок;
- сохранять прежнее финансовое поведение активных и исторических заявок;
- исключить пересчёт истории по текущему env.

### 2.3 Миграция существующих данных

Миграция `20260724000000_add_commercial_mode`:

1. создаёт PostgreSQL enum `CommercialMode`;
2. добавляет обязательное поле в `Order`;
3. добавляет обязательное поле в `PlannedOrder`;
4. относит все существующие записи к `PAID_MOCK`;
5. создаёт аналитические индексы.

Исторические строки нельзя автоматически считать бесплатными: до появления режима система всегда использовала mock-коммерческий контур.

### 2.4 Главный инвариант

```text
commercialMode заявки неизменяем после создания
```

Смена:

```env
COMMERCIAL_MODE=FREE_PILOT
```

на:

```env
COMMERCIAL_MODE=PAID_MOCK
```

влияет только на новые заявки.

Обновлять `commercialMode` существующей записи административным скриптом запрещено без отдельной миграции данных, финансовой сверки и аудита.

## 3. Пользователи и авторизация

### 3.1 `User`

Единая учётная запись клиента, мастера и оператора.

Ключевые поля:

- `id` — UUID;
- `phone` — уникальный номер;
- `name`;
- `defaultAddress` — старое строковое поле;
- `role` — `CLIENT` или `OPERATOR`;
- `createdAt`.

Отдельной роли `MASTER` в `UserRole` нет. Возможность выполнять действия мастера определяется:

- наличием `MasterProfile`;
- статусом профиля;
- текущей блокировкой;
- наличием оффера или назначением на конкретную заявку.

### 3.2 `SmsCode`

Хранит:

- телефон;
- шестизначный код;
- срок действия;
- количество неуспешных попыток;
- время использования;
- время создания.

Индекс:

```text
(phone, createdAt)
```

Текущие риски:

- код хранится открытым текстом;
- нет назначения кода (`LOGIN`, `CHANGE_PHONE` и т.п.);
- нет автоматического retention/cleanup;
- нет device/session модели;
- нет refresh token и отзыва отдельных сессий.

## 4. Категории и верификация мастеров

### 4.1 `Category`

Справочник услуг:

- `id`;
- уникальный `slug`;
- отображаемое `name`.

Категория связана с:

- компетенциями мастеров;
- срочными заявками;
- плановыми заявками.

### 4.2 `MasterProfile`

Анкета и рабочее состояние мастера.

Ключевые поля:

- уникальный `userId`;
- ФИО;
- ИИН;
- район;
- стаж;
- статус проверки;
- причина отклонения;
- `priorityPenaltyUntil`;
- `blockedUntil`;
- timestamps.

Статусы:

```text
PENDING_REVIEW
NEEDS_INFO
ACTIVE
REJECTED
```

ИИН сейчас хранится строкой без шифрования на уровне приложения. Это персональные данные повышенной чувствительности.

### 4.3 `MasterCategory`

Many-to-many связь профиля мастера с категориями.

Составной первичный ключ:

```text
(masterProfileId, categoryId)
```

### 4.4 `MasterDocument`

Метаданные документа:

- профиль;
- тип `ID_CARD` или `QUALIFICATION`;
- относительный путь;
- исходное имя;
- MIME;
- размер;
- время создания.

Бинарное содержимое находится в файловом хранилище.

### 4.5 `VerificationDecision`

История решений оператора:

- профиль;
- оператор;
- решение `APPROVE`, `REJECT`, `REQUEST_INFO`;
- комментарий;
- время.

Запись используется как аудит решения, но в модели пока нет отдельной защиты от изменения/удаления оператором базы данных.

## 5. Срочные заявки

### 5.1 `Order`

Главная сущность срочного режима.

#### Участники

- `clientId` — обязательный клиент;
- `masterId` — nullable до принятия.

#### Задача и адрес

- категория;
- описание;
- адрес;
- район;
- подъезд, этаж, квартира, комментарий;
- PostGIS `location`.

#### Поиск

- `status`;
- `wave`;
- `searchAttempt`.

#### Коммерческий контекст

- `commercialMode`;
- `calloutPrice`;
- `serviceFee`;
- `workPrice`;
- `workComment`.

В `FREE_PILOT`:

- `calloutPrice` и `serviceFee` сохраняют номинальный расчёт в БД;
- публичное HTTP/Socket.IO-представление показывает их как `0`;
- `workPrice` хранит согласованную цену работ;
- платформа не принимает `workPrice`: расчёт происходит напрямую между сторонами.

Следовательно, эти поля нельзя без фильтра использовать как фактическую выручку или платёжный оборот.

#### Жизненный цикл

- `acceptedAt`;
- `onSiteAt`;
- `priceProposedAt`;
- `completedAt`;
- `closedAt`;
- `cancelReason`;
- `createdAt`, `updatedAt`.

Индексы:

```text
(clientId, status)
(masterId, status)
(commercialMode, createdAt)
```

Статусы:

```text
CREATED
SEARCHING
ACCEPTED
MASTER_ON_WAY
INSPECTION
AWAITING_PRICE_CONFIRM
IN_PROGRESS
DONE
CLOSED
NO_MASTERS
CANCELLED_BY_CLIENT
CANCELLED_BY_MASTER
DISPUTE
```

Фактическое поведение:

- отмена мастером из `ACCEPTED`/`MASTER_ON_WAY` возвращает заявку в `SEARCHING`;
- `CANCELLED_BY_MASTER` существует в enum, но не используется как конечное состояние этого сценария;
- спор хранится отдельной сущностью и не обязательно переводит поле `Order.status` в `DISPUTE`.

### 5.2 `OrderOffer`

Фиксирует предложение конкретному мастеру:

- заявка;
- мастер;
- волна;
- попытка поиска;
- время отправки;
- время ответа;
- outcome.

Outcome:

```text
PENDING
ACCEPTED
LOST
EXPIRED
```

Критическое ограничение:

```text
unique(orderId, masterUserId, attempt)
```

Оно предотвращает дублирование оффера одному мастеру в одной попытке поиска.

### 5.3 `MasterPresence`

Текущее online/geolocation-состояние мастера:

- `masterUserId` — первичный ключ;
- `isOnline`;
- `lastSeenAt`;
- PostGIS `location`.

История перемещений не хранится.

### 5.4 `OrderPhoto`

Хранит относительный путь изображения и время создания.

Связь удаляется каскадно вместе с заявкой. Физический файл при этом автоматически не гарантированно удаляется из хранилища.

## 6. Плановые заявки

### 6.1 `PlannedOrder`

Основные группы полей:

- клиент и категория;
- описание и адрес;
- предполагаемый бюджет;
- `slotStart`, `slotEnd`;
- статус;
- `commercialMode`;
- назначенный мастер;
- выбранный отклик;
- `workPrice`;
- причина отмены;
- timestamps жизненного цикла.

Статусы:

```text
CREATED
PUBLISHED
MASTER_SELECTED
CONFIRMED
IN_PROGRESS
DONE
CLOSED
EXPIRED
CANCELLED_BY_CLIENT
CANCELLED_BY_MASTER
DISPUTE
```

Фактическое поведение:

- сервис создаёт заявку сразу в `PUBLISHED`;
- `CREATED` пока не является отдельным пользовательским этапом;
- отмена мастером возвращает подтверждённую заявку в `PUBLISHED`, а не в `CANCELLED_BY_MASTER`.

Индексы:

```text
(clientId, status)
(masterId, status)
(categoryId, status)
(commercialMode, createdAt)
```

### 6.2 `PlannedOrderBid`

Отклик мастера:

- заявка;
- мастер;
- цена;
- текстовый срок `term`;
- комментарий;
- время создания.

Ограничение:

```text
unique(plannedOrderId, masterUserId)
```

Один мастер может оставить только один отклик на заявку.

`PlannedOrder.selectedBidId`:

- nullable;
- уникален;
- указывает на выбранный отклик.

Сервис дополнительно проверяет, что выбранный отклик принадлежит заявке.

### 6.3 Коммерческое поведение отклика

Для `PlannedOrder.commercialMode=FREE_PILOT`:

- баланс lead-кредитов не проверяется;
- `LeadCreditAccount.balance` не уменьшается;
- `LeadCreditTransaction(SPEND)` не создаётся;
- при отмене клиентом `REFUND` не создаётся.

Для `PAID_MOCK` сохраняется прежнее списание одного кредита.

Решение принимается по режиму конкретной заявки, поэтому в одной ленте могут одновременно существовать бесплатные и платные записи.

### 6.4 `PlannedOrderPhoto`

Относительные пути изображений плановой заявки.

## 7. Адреса

### `Address`

Нормализованный адрес пользователя:

- `userId`;
- метка;
- адрес;
- подъезд, этаж, квартира, комментарий;
- `lat`, `lng` как `Float`;
- `isDefault`;
- время создания.

Индекс:

```text
(userId)
```

Ограничение «не более одного адреса по умолчанию на пользователя» отсутствует на уровне БД. Сейчас его обеспечивает сервисная логика.

Рекомендуемое усиление PostgreSQL:

```sql
CREATE UNIQUE INDEX ... ON "Address"("userId") WHERE "isDefault" = true;
```

## 8. Платежи срочного режима

### 8.1 `PaymentTransaction`

Журнал операций:

- `orderId`;
- тип `HOLD`, `CAPTURE`, `VOID`;
- сумма;
- статус `PENDING`, `SUCCEEDED`, `FAILED`;
- reference провайдера;
- время.

Индекс:

```text
(orderId, type)
```

Уникального DB-ограничения на одну операцию каждого типа нет. Идемпотентность сейчас зависит от provider/service-логики.

### 8.2 `Accrual`

Начисление компенсации мастеру за выезд.

Ключевой инвариант:

```text
orderId @unique
```

Это ограничивает начисление одной записью на срочную заявку.

Тип:

```text
CALLOUT_COMPENSATION
```

### 8.3 Поведение по режиму

`FREE_PILOT` заявка не должна создавать:

- `PaymentTransaction` для HOLD/CAPTURE/VOID/refund;
- `Accrual`;
- увеличение `MasterWalletAccount.balance`.

`PAID_MOCK` продолжает использовать эти таблицы.

Финансовые таблицы не удаляются: они нужны для тестового коммерческого режима и будущего `PAID_LIVE`.

## 9. Lead-кредиты

### 9.1 `LeadCreditAccount`

Текущий баланс мастера. Первичный ключ:

```text
masterUserId
```

### 9.2 `LeadCreditTransaction`

Типы:

```text
PURCHASE
SPEND
REFUND
PENALTY
```

Поля `bidId` и `purchaseId` хранят идентификаторы, но не оформлены Prisma relations/foreign keys.

### 9.3 `LeadCreditPurchase`

Хранит:

- мастера;
- количество кредитов;
- цену;
- статус платёжной операции;
- reference провайдера;
- время.

В глобальном `FREE_PILOT` новые покупки блокируются. Для исторической `PAID_MOCK` плановой заявки transactional-решение о списании всё равно определяется `PlannedOrder.commercialMode`.

## 10. Кошелёк и выводы

### 10.1 `MasterWalletAccount`

Текущий доступный баланс мастера. Первичный ключ:

```text
masterUserId
```

### 10.2 `WithdrawalRequest`

Хранит:

- мастера;
- сумму;
- статус `PENDING`, `PAID`, `FAILED`;
- reference провайдера;
- время запроса;
- время выплаты.

В глобальном `FREE_PILOT`:

- баланс API показывается как `0`;
- история выводов скрывается;
- создание нового вывода запрещено;
- таблица не очищается автоматически и может содержать старые mock-записи тестовой среды.

Перед production-пилотом нельзя переносить тестовые wallet/withdrawal строки как реальные обязательства.

## 11. Споры

### `Dispute`

Спор относится к:

- `orderId`;
- либо `plannedOrderId`.

Основные поля:

- пользователь, открывший спор;
- роль открывшего `CLIENT`/`MASTER`;
- причина;
- массив путей доказательств;
- пояснение второй стороны;
- статус `OPEN`/`RESOLVED`;
- `refundServiceFee`;
- `penalizeMaster`;
- текст решения;
- оператор и время разрешения.

Индексы:

```text
(orderId)
(plannedOrderId)
(status)
```

Текущие ограничения схемы:

- нет check constraint «заполнен ровно один target»;
- нет уникального ограничения «один спор на заявку»;
- сервис обрабатывает Prisma `P2002` при повторном открытии, но без unique index этот путь не гарантирован;
- массив `evidenceDocIds` хранит пути без отдельной таблицы метаданных и checksum.

В `FREE_PILOT` разрешение спора принудительно сохраняет `refundServiceFee=false`, потому что платформа не принимала сбор.

## 12. Отзывы

### `Review`

Поля:

- `orderId` либо `plannedOrderId`;
- клиент-автор;
- мастер-получатель;
- рейтинг;
- комментарий;
- время.

`orderId` и `plannedOrderId` по отдельности уникальны, что обеспечивает не более одного отзыва каждого типа на target.

Однако нет check constraint, запрещающего одновременно заполнить оба поля или оставить оба `null`.

## 13. Санкции и отмены мастера

### `MasterCancellation`

Аудит отмен:

- `masterUserId`;
- `orderType`: `URGENT` или `PLANNED`;
- `orderId` строкой;
- время.

Связь с заявкой полиморфная и не имеет foreign key. Поэтому удаление/изменение заявки может оставить audit-запись без проверяемого target.

Индекс:

```text
(masterUserId, createdAt)
```

## 14. Финансовые инварианты бесплатного пилота

Для любой записи с `commercialMode=FREE_PILOT` должны выполняться условия:

### Срочная заявка

```text
PaymentTransaction по orderId = 0 строк
Accrual по orderId = 0 строк
компенсация не увеличивает MasterWalletAccount.balance
```

### Плановая заявка

```text
отклик создаётся без LeadCreditAccount
SPEND по bidId = 0 строк
REFUND по выбранному bidId = 0 строк
```

### Глобальные операции

```text
новые LeadCreditPurchase запрещены
новые WithdrawalRequest запрещены
```

Номинальные суммы в `Order`, `PlannedOrder`, `PlannedOrderBid` не нарушают инвариант: это аналитические/договорные значения, а не доказательство платёжной операции.

## 15. Аналитика по режиму

Все отчёты по выручке, платежам и unit economics обязаны фильтровать режим.

Пример распределения заявок:

```sql
SELECT "commercialMode", COUNT(*)
FROM "Order"
GROUP BY "commercialMode";
```

Активные срочные заявки:

```sql
SELECT "commercialMode", status, COUNT(*)
FROM "Order"
WHERE status NOT IN ('CLOSED', 'CANCELLED_BY_CLIENT', 'CANCELLED_BY_MASTER')
GROUP BY "commercialMode", status;
```

Проверка финансовых нарушений пилота:

```sql
SELECT o.id, COUNT(pt.id) AS payment_count, COUNT(a.id) AS accrual_count
FROM "Order" o
LEFT JOIN "PaymentTransaction" pt ON pt."orderId" = o.id
LEFT JOIN "Accrual" a ON a."orderId" = o.id
WHERE o."commercialMode" = 'FREE_PILOT'
GROUP BY o.id
HAVING COUNT(pt.id) > 0 OR COUNT(a.id) > 0;
```

Ожидаемый результат — 0 строк.

Нельзя считать:

```text
SUM(Order.calloutPrice)
SUM(Order.serviceFee)
SUM(Order.workPrice)
```

фактической выручкой без связи с подтверждёнными финансовыми операциями и фильтра по режиму.

## 16. Обязательные инварианты для тестов

1. Режим заявки фиксируется один раз при создании.
2. Изменение env не меняет поведение существующей заявки.
3. Исторические записи после миграции имеют `PAID_MOCK`.
4. Один активный срочный заказ клиента.
5. Один активный срочный заказ назначенного мастера.
6. Один оффер мастеру на одну попытку поиска.
7. Один отклик мастера на плановую заявку.
8. Выбранный отклик принадлежит выбираемой заявке.
9. Не более одного начисления на срочную заявку.
10. Финансовые таблицы не изменяются для `FREE_PILOT`.
11. Статус изменяется только из разрешённого предыдущего состояния.
12. Auto-close не выполняется при открытом споре.
13. Точный адрес плановой заявки скрыт от неназначенного мастера.
14. Только участники и оператор получают защищённые файлы.
15. Один отзыв на заявку.
16. `order:status` и HTTP маскируют номинальные суммы бесплатной заявки.
17. Историческая `PAID_MOCK` плановая заявка требует кредит даже при текущем env `FREE_PILOT`.

## 17. Известные пробелы схемы

### P0/P1 до коммерческого production

- unique/check constraints для `Dispute`;
- exactly-one-target constraint для `Review`;
- частичный unique index одного default-адреса;
- DB-level idempotency для платёжных операций;
- audit log изменения критических полей;
- запрет обновления `commercialMode` через административные инструменты;
- retention и удаление персональных данных;
- шифрование/токенизация ИИН и чувствительных документов;
- checksum, MIME sniffing и object-storage metadata для файлов.

### Дальнейшее развитие

- таблица истории статусов `OrderStatusHistory`/event log;
- optimistic version или sequence для REST/Socket.IO;
- структурированный срок выполнения вместо свободного `PlannedOrderBid.term`;
- версия тарифа, по которой рассчитаны номинальные цены;
- ledger-подход вместо изменяемых balance-полей;
- отдельные provider operation/idempotency keys;
- soft delete и legal hold для споров/финансового аудита.

## 18. Правила изменения модели

Любое изменение следующих элементов требует миграции, тестов и обновления документации в одном PR:

- enum статусов;
- `CommercialMode`;
- финансовые таблицы;
- unique/check/index constraints;
- target-связи споров и отзывов;
- PostGIS-поля;
- retention персональных данных;
- структура Socket.IO/HTTP-представления коммерческих полей.

Миграции production нельзя заменять `prisma db push`. Для rollout используется:

```bash
pnpm --filter api exec prisma migrate deploy
```
