# WebSocket-события MasterQala.kz

Документ описывает фактически реализованный realtime-контракт Socket.IO.

## 1. Подключение

Backend использует Socket.IO через NestJS `WebSocketGateway`.

Токен передаётся в handshake:

```ts
io(API_ORIGIN, {
  auth: { token: accessToken },
});
```

При подключении backend:

1. проверяет JWT;
2. извлекает `sub` как идентификатор пользователя;
3. присоединяет socket к комнате `user:<userId>`;
4. отклоняет handshake с `connect_error`, если токен невалиден.

События пользователю отправляются во все его активные вкладки и устройства, подключённые к комнате.

## 2. Общие правила клиента

- REST остаётся источником истины;
- WebSocket используется для ускорения интерфейса, а не как единственное хранилище состояния;
- после reconnect frontend должен повторно запросить активную срочную заявку, плановые заявки и профиль;
- события сейчас не имеют глобального `eventId`, версии, sequence number или acknowledgement;
- повторная доставка и доставка вне ожидаемого порядка должны считаться возможными;
- обработчики должны быть идемпотентными: применять событие только к совпадающему `orderId` или `plannedOrderId`;
- неизвестные поля и неизвестные события должны безопасно игнорироваться.

## 3. События client → server

### 3.1 `presence:online`

Мастер включает режим доступности.

Payload:

```json
{
  "lat": 51.128,
  "lng": 71.430
}
```

Backend сохраняет:

- `isOnline = true`;
- текущие координаты;
- `lastSeenAt`.

Некорректный payload без числовых координат игнорируется без ошибки.

### 3.2 `presence:offline`

Мастер выключает доступность.

Payload отсутствует.

Backend устанавливает `isOnline = false`.

При разрыве socket-соединения backend также переводит пользователя офлайн. Это означает, что при нескольких одновременных устройствах отключение одного socket сейчас может сделать пользователя офлайн, даже если другое соединение осталось активно.

### 3.3 `geo:update`

Обновление геопозиции мастера.

Payload:

```json
{
  "lat": 51.128,
  "lng": 71.430
}
```

Backend:

1. обновляет `MasterPresence.location` и `lastSeenAt`;
2. если мастер едет по активной срочной заявке, рассчитывает ETA;
3. отправляет клиенту `master:location`.

Ретрансляция выполняется только для статусов:

- `ACCEPTED`;
- `MASTER_ON_WAY`.

## 4. События server → client: срочные заявки

### 4.1 `offer:new`

Получатель: мастер-кандидат текущей волны.

```json
{
  "orderId": "uuid",
  "category": "Сантехника",
  "description": "Течёт кран",
  "district": "Есиль",
  "distanceKm": 2.4,
  "compensation": 1200,
  "deadline": "2026-07-23T18:20:00.000Z",
  "wave": 1
}
```

Поля:

- `compensation` — номинальная сумма выезда мастеру: `calloutPrice - serviceFee`;
- `deadline` — момент окончания текущего оффера;
- `distanceKm` — оценка дорожного расстояния с коэффициентом маршрутизации;
- `wave` — номер волны 1–3.

В `FREE_PILOT` поле `compensation` является аналитическим и не означает автоматическую выплату платформой. В интерфейсе его следует либо переименовать в «ориентир выезда», либо скрыть до утверждения пилотной модели расчётов.

### 4.2 `offer:closed`

Получатель: мастер, которому ранее был отправлен оффер.

```json
{
  "orderId": "uuid",
  "reason": "Заявку принял другой мастер"
}
```

Возможные причины формируются backend как текст:

- заявку принял другой мастер;
- время на принятие истекло;
- клиент отменил заявку;
- поиск завершён.

Frontend не должен строить бизнес-логику на точном русском тексте. До публичного запуска необходимо добавить машинное поле `reasonCode`.

### 4.3 `order:status`

Получатели:

- клиент заявки;
- назначенный мастер, если он существует.

```json
{
  "orderId": "uuid",
  "status": "AWAITING_PRICE_CONFIRM",
  "wave": 2,
  "master": {
    "id": "uuid",
    "name": "Мастер",
    "phone": "+77000000000"
  },
  "workPrice": 15000,
  "workComment": "Замена смесителя",
  "cancelReason": null,
  "calloutPrice": 2500,
  "priceProposedAt": "2026-07-23T18:30:00.000Z"
}
```

Событие отправляется после значимых переходов state machine, изменения волны и завершения поиска.

После получения frontend должен:

1. проверить совпадение `orderId`;
2. обновить видимые поля;
3. при неизвестном или терминальном переходе повторно запросить `GET /orders/:id`;
4. не вычислять окончательную доступность кнопок только по локальному состоянию.

### 4.4 `master:location`

Получатель: клиент активной срочной заявки.

```json
{
  "orderId": "uuid",
  "lat": 51.128,
  "lng": 71.430,
  "etaMinutes": 9
}
```

Событие содержит текущие координаты мастера и приблизительное ETA до точки заказа.

Требования frontend:

- обновлять маркер без полной перезагрузки карты;
- не сохранять историю координат в браузере дольше активной заявки;
- прекращать отображение после выхода заявки из `ACCEPTED`/`MASTER_ON_WAY`;
- учитывать, что ETA может быть `0`, если координаты заказа отсутствуют или запрос расстояния не вернул строку.

## 5. События server → client: плановые заявки

### 5.1 `bid:new`

Получатель: клиент опубликованной плановой заявки.

```json
{
  "plannedOrderId": "uuid",
  "bidsCount": 3
}
```

Событие сообщает только новое количество откликов. Для получения данных мастера, цены и срока клиент повторно запрашивает заявку.

### 5.2 `bid:selected`

Получатель: выбранный мастер.

```json
{
  "plannedOrderId": "uuid"
}
```

После события мастер должен загрузить подробности заявки через REST и подтвердить или отклонить выбор.

### 5.3 `bid:closed`

Получатель: мастер, отклик которого не выбран.

```json
{
  "plannedOrderId": "uuid",
  "reason": "Выбран другой мастер"
}
```

До стабильного контракта требуется добавить `reasonCode`.

### 5.4 `planned:status`

Получатели:

- клиент заявки;
- назначенный мастер.

```json
{
  "plannedOrderId": "uuid",
  "status": "CONFIRMED",
  "workPrice": 20000,
  "cancelReason": null,
  "slotStart": "2026-07-25T08:00:00.000Z",
  "slotEnd": "2026-07-25T11:00:00.000Z",
  "master": {
    "id": "uuid",
    "name": "Мастер",
    "phone": "+77000000000"
  }
}
```

Для клиента контакт мастера редактируется до статуса `CONFIRMED`. Для назначенного мастера backend отправляет полные доступные данные.

## 6. Reconnect и восстановление состояния

Рекомендуемый алгоритм frontend после `connect` или `reconnect`:

1. `GET /users/me`;
2. если пользователь клиент — `GET /orders/active` и `GET /planned-orders/mine`;
3. если пользователь активный мастер — `GET /master/active-order` и `GET /planned-orders/feed`;
4. открыть детали текущей заявки через REST;
5. только после синхронизации применять новые realtime-события.

Нельзя считать отсутствие события доказательством отсутствия изменения: пользователь мог быть офлайн, браузер мог выгрузить PWA, а Socket.IO-событие не является долговечным сообщением.

## 7. Безопасность realtime

Текущие меры:

- JWT проверяется во время handshake;
- socket присоединяется только к комнате собственного пользователя;
- клиент не передаёт произвольный `userId` для подписки;
- координаты проверяются на тип `number`.

Недостающие меры до production:

1. ограничить CORS конкретными origin;
2. проверять роль и активный статус мастера для `presence:online` и `geo:update`;
3. проверять диапазоны координат: `lat [-90, 90]`, `lng [-180, 180]`;
4. добавить rate limit на геообновления;
5. вести аудит подключения и аномальных обновлений;
6. поддержать несколько socket-соединений одного пользователя через счётчик активных соединений;
7. добавить версию payload и машинные коды причин закрытия;
8. исключить чувствительные поля из событий, если получатель не имеет права на них;
9. определить частоту и точность геолокации в политике конфиденциальности.

## 8. Поведение в бесплатном пилоте

Realtime-архитектура не меняется. Отключение монетизации влияет только на семантику финансовых полей:

- `offer:new.compensation` — номинальный ориентир, не обещание выплаты;
- `order:status.calloutPrice` и `workPrice` — зафиксированные договорённости сторон;
- плановые отклики бесплатны, поэтому realtime не зависит от баланса lead-кредитов;
- никаких отдельных событий `payment:*`, `wallet:*` или `lead-credit:*` в первой версии не требуется.

## 9. Обязательные тесты контракта

- невалидный JWT получает `connect_error`;
- пользователь не может подписаться на комнату другого пользователя;
- `offer:new` получает только кандидат созданного `OrderOffer`;
- после принятия проигравшие получают `offer:closed`;
- клиент и мастер получают одинаковый новый статус заявки;
- клиент получает геолокацию только назначенного мастера своей заявки;
- после reconnect состояние восстанавливается через REST;
- невалидные координаты не изменяют presence;
- plan bid события отправляются только связанным пользователям;
- в `FREE_PILOT` события не подразумевают создание финансовых транзакций.
