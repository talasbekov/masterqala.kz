# Безопасность загрузки файлов

## Назначение

Этот документ фиксирует текущий security baseline для файлов бесплатного пилота MasterQala.kz.

Он покрывает:

- фотографии срочных и плановых заявок;
- документы заявки мастера;
- доказательства в споре;
- локальное хранение файлов API;
- ownership, TTL и одноразовую привязку фотографий заявки.

## Разрешённые форматы

| Контекст | Форматы | Максимальный размер |
|---|---|---:|
| Фото заявки | JPEG, PNG | 10 МБ |
| Доказательство спора | JPEG, PNG | 10 МБ |
| Документ мастера | JPEG, PNG, PDF | 10 МБ |

Тип определяется не только по заголовку `Content-Type`, переданному клиентом.

API сверяет одновременно:

1. фактическую сигнатуру первых байтов;
2. заявленный MIME;
3. последнее расширение исходного имени;
4. allowlist конкретного endpoint.

Проверяемые сигнатуры:

- JPEG: `FF D8 FF`;
- PNG: `89 50 4E 47 0D 0A 1A 0A`;
- PDF: `%PDF-`.

Несоответствие любого слоя возвращает `400 Bad Request`.

## Имена и пути

Исходное имя пользователя не используется как имя файла на диске.

Файл сохраняется как:

```text
<random UUID>.<canonical extension>
```

Примеры:

```text
7632082e-cd46-441a-b184-bd6cc0a42e09.jpg
28fe54a5-bf55-4f5d-98df-44f351358e94.pdf
```

Исходное имя документа мастера сохраняется только как метаданные и предварительно очищается:

- удаляются `/` и `\`;
- удаляются управляющие символы;
- удаляются bidi override/isolate символы;
- опасные символы заменяются;
- длина ограничивается 180 символами;
- пустое имя заменяется на `file.<ext>`.

## Хранение

`LocalDiskStorage`:

- создаёт каталог с режимом `0700`;
- создаёт файлы с режимом `0600`;
- использует `writeFile(..., { flag: "wx" })`;
- не перезаписывает существующий файл;
- принимает только короткое alphanumeric расширение;
- блокирует пустой путь и path traversal;
- умеет безопасно проверять наличие и удалять файл.

`UPLOAD_DIR` должен находиться вне каталога со статической раздачей frontend и не должен напрямую публиковаться Nginx.

Файл выдаётся только через авторизованный API endpoint после проверки участника заявки или роли оператора.

## PendingUpload: владелец и TTL

Каждая фотография, загруженная через `POST /api/v1/uploads`, получает запись `PendingUpload`:

```text
id
userId
path
mimeType
sizeBytes
expiresAt
consumedAt
createdAt
```

Правила:

- `userId` берётся только из проверенного JWT;
- `path` уникален;
- default TTL — 24 часа;
- `UPLOAD_TTL_HOURS` допускает значения от 1 до 168;
- до привязки `consumedAt = null`;
- истёкший или уже использованный upload нельзя привязать;
- API не сообщает, является ли путь чужим, истёкшим или использованным.

Ответ upload endpoint содержит `expiresAt`, чтобы клиент мог своевременно обновить черновик.

## Привязка фотографий к заявкам

`photoPaths` ограничен:

- максимум пятью значениями;
- уникальными значениями;
- форматом `UUID.jpg` или `UUID.png`;
- существующим файлом в configured storage;
- upload-записью текущего пользователя;
- неистёкшим TTL;
- состоянием `consumedAt = null`.

Произвольный путь, PDF, path traversal, отсутствующий, чужой, истёкший или повторно используемый upload не могут быть привязаны к заявке.

## Атомарное одноразовое consume

API guard выполняет предварительную проверку и возвращает понятный `400`.

Окончательное правило обеспечивает PostgreSQL:

- `BEFORE INSERT` trigger на `OrderPhoto` получает `clientId` срочной заявки;
- `BEFORE INSERT` trigger на `PlannedOrderPhoto` получает `clientId` плановой заявки;
- trigger одним условным `UPDATE` выставляет `PendingUpload.consumedAt`;
- `UPDATE` проходит только для владельца, неистёкшей и ещё не использованной записи;
- если строка не обновлена, вставка фотографии завершается ошибкой;
- consume выполняется в той же транзакции, что и создание заявки и `OrderPhoto`/`PlannedOrderPhoto`;
- rollback заявки автоматически откатывает `consumedAt`.

Это закрывает гонку между API precheck и записью фотографий: два параллельных запроса не смогут успешно использовать один путь.

## Удаление истёкших uploads

`PendingUploadsService` регистрирует pg-boss cron:

```text
17 * * * *
```

Каждый час cleanup:

1. выбирает до 100 истёкших записей с `consumedAt = null`;
2. удаляет файл из storage;
3. удаляет запись `PendingUpload` с повторной проверкой состояния;
4. логирует ошибку и продолжает со следующей записью.

Использованные uploads не удаляются этим cleanup, потому что их файл уже является частью заявки.

При отключённом pg-boss автоматический cleanup не выполняется; для production `PGBOSS_DISABLED` использовать нельзя.

## Откат при ошибке БД

Для регистрации pending upload:

1. файл проходит validation;
2. файл сохраняется;
3. создаётся `PendingUpload`;
4. если создание записи упало, файл удаляется.

Для документов мастера и доказательств спора действует аналогичный compensating cleanup.

Ошибка удаления orphan-файла логируется и не маскирует исходную ошибку БД.

Multer использует memory storage, поэтому временные файлы до validation на диске не создаются.

## Content-Type при скачивании

API не доверяет историческому `mimeType` из БД для HTTP-ответа.

Content-Type определяется по каноническому сохранённому расширению:

- `.jpg` → `image/jpeg`;
- `.png` → `image/png`;
- `.pdf` → `application/pdf`.

Неизвестное расширение не выдаётся.

PDF документов мастера скачивается как `attachment`, а не открывается inline.

## Что этот baseline не решает

Magic-byte validation не является антивирусом.

Она не гарантирует обнаружение:

- polyglot-файлов;
- вредоносного содержимого внутри корректного PDF;
- PDF JavaScript и embedded files;
- специально подготовленных parser exploits;
- image decompression bombs;
- новых сигнатур вредоносного ПО.

До загрузки высокого риска необходимо добавить асинхронный quarantine pipeline:

1. сохранить файл в закрытый quarantine storage;
2. создать запись со статусом `PENDING_SCAN`;
3. проверить ClamAV или внешним malware scanner;
4. для PDF применить Content Disarm and Reconstruction при необходимости;
5. только после `CLEAN` переносить в доступное storage;
6. `INFECTED` удалять и фиксировать security audit event.

## Ограничения эксплуатации

- размер файла ограничивается API и должен дополнительно ограничиваться reverse proxy;
- memory storage означает расход RAM до 10 МБ на одновременный upload;
- для пилота следует ограничить upload concurrency на Nginx;
- local disk подходит только для одного API instance;
- при нескольких replicas нужен S3-compatible private bucket или общий защищённый storage;
- файловая система и PostgreSQL не являются одной транзакционной системой;
- компенсирующий cleanup закрывает обычные ошибки, но аварийное завершение процесса между disk write и DB insert может оставить orphan-файл;
- cleanup обрабатывает 100 файлов за запуск, поэтому большой backlog очищается несколькими проходами;
- использованные `PendingUpload` записи пока сохраняются для аудита без отдельной retention policy;
- резервная копия БД должна быть согласована с резервной копией файлов.

## Smoke-проверки staging

### MIME spoofing

PNG с заголовком `Content-Type: image/jpeg` должен вернуть `400`.

### Двойное расширение

JPEG с именем `photo.jpg.exe` должен вернуть `400`.

### Path traversal

Создание заявки с:

```json
{
  "photoPaths": ["../../etc/passwd"]
}
```

должно завершиться validation error.

### Чужой upload

Пользователь B не должен создать заявку с путём, который загрузил пользователь A. Запись A должна остаться с `consumedAt = null`.

### Одноразовое использование

После успешного создания заявки повторная привязка того же пути должна вернуть `400`, а в таблице фотографий должна остаться одна запись.

### Истёкший upload

После `expiresAt` путь должен возвращать `400` и не создавать фотографию заявки.

### Атомарность

Прямая попытка вставить второй `OrderPhoto` или `PlannedOrderPhoto` с уже потреблённым путём должна быть отклонена PostgreSQL trigger.

### Content-Type

- PNG фотографии должен возвращаться как `image/png`;
- JPEG — как `image/jpeg`;
- PDF документа мастера — как `application/pdf` и `attachment`.

### Orphan rollback

При искусственной ошибке создания `PendingUpload`, `MasterDocument` или обновления `Dispute` файл не должен оставаться в `UPLOAD_DIR`.
