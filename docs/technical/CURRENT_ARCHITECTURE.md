# Текущая архитектура MasterQala.kz

Статус документа: описание фактической реализации первой версии по состоянию репозитория.

## 1. Общая схема

MasterQala.kz реализован как TypeScript-монорепозиторий на `pnpm`:

- `apps/api` — backend на NestJS;
- `apps/web` — единое React PWA для клиента, мастера и оператора;
- `packages/ui` — общие UI-компоненты;
- PostgreSQL 16 + PostGIS — основная база данных и географические запросы;
- pg-boss — фоновые задания и отложенные таймауты;
- Socket.IO — realtime-события;
- локальное файловое хранилище — документы мастеров и фотографии заявок;
- mock payment provider — имитация платежей, пополнений, выплат и возвратов.

```text
React PWA
  ├─ HTTP/JSON ───────────────► NestJS API
  └─ Socket.IO + JWT ─────────► RealtimeGateway
                                  │
NestJS modules                   ├─ PostgreSQL/PostGIS
  ├─ auth                        ├─ pg-boss
  ├─ masters                     ├─ local uploads
  ├─ orders                      └─ mock payments
  ├─ planned-orders
  ├─ disputes
  ├─ reviews
  ├─ wallet / lead credits
  └─ operator
```

## 2. Backend

### 2.1 Технологии

Фактический backend-стек:

- NestJS 10;
- TypeScript;
- Prisma 5;
- PostgreSQL/PostGIS;
- class-validator/class-transformer;
- JWT;
- Socket.IO;
- pg-boss;
- Jest + Supertest.

### 2.2 Модульность

Основные домены разделены на NestJS-модули. Ключевая бизнес-логика находится в сервисах, а контроллеры выполняют роль HTTP-адаптеров.

Критические переходы состояния выполняются через атомарный `updateMany` с условием по текущему статусу. Если строка не обновлена, API возвращает конфликт. Этот механизм защищает от повторных нажатий и гонок, например когда несколько мастеров одновременно принимают одну заявку.

### 2.3 Авторизация

- вход выполняется по номеру телефона и SMS-коду;
- в development SMS-код выводится в лог API;
- после подтверждения выдаётся JWT;
- HTTP endpoints защищаются `JwtAuthGuard`;
- Socket.IO проверяет JWT во время handshake;
- после подключения пользователь входит в персональную комнату `user:<userId>`.

В Prisma сейчас существуют системные роли `CLIENT` и `OPERATOR`. Мастер не является отдельной ролью пользователя: возможности мастера определяются наличием `MasterProfile` и его статусом.

### 2.4 Данные

Основные сущности:

- `User`;
- `MasterProfile`, категории и документы мастера;
- `Order` и `OrderOffer` для срочного режима;
- `PlannedOrder` и `PlannedOrderBid` для планового режима;
- `PaymentTransaction`;
- `Accrual`;
- `LeadCreditAccount` и журнал операций;
- `MasterWalletAccount` и заявки на вывод;
- `Dispute`;
- `Review`;
- сохранённые адреса;
- присутствие и геопозиция мастера.

Денежные значения хранятся целыми числами в тенге. Географические точки срочных заявок и присутствия мастеров хранятся как `geography(Point, 4326)`.

## 3. Срочный режим

### 3.1 Создание

Клиент получает предварительный расчёт через `POST /orders/preview`, затем создаёт заявку через `POST /orders`.

Backend:

1. проверяет категорию;
2. запрещает вторую активную срочную заявку;
3. ищет потенциальных мастеров и рассчитывает выезд;
4. создаёт заказ и географическую точку;
5. создаёт HOLD на полную стоимость выезда;
6. переводит `CREATED → SEARCHING`;
7. ставит первую волну матчинга в pg-boss.

### 3.2 Матчинг

Используются три последовательные волны:

| Волна | Радиус | Ожидание |
|---|---:|---:|
| 1 | 3 км | 60 секунд |
| 2 | 6 км | 60 секунд |
| 3 | 10 км | 90 секунд |

Оффер получают только подходящие активные мастера. Принятие защищено атомарным переходом `SEARCHING → ACCEPTED`. Первый мастер выигрывает; остальные получают `offer:closed`.

### 3.3 Выполнение

Основная цепочка:

`CREATED → SEARCHING → ACCEPTED → MASTER_ON_WAY → INSPECTION → AWAITING_PRICE_CONFIRM → IN_PROGRESS → DONE → CLOSED`.

- на подтверждение цены клиенту даётся 15 минут;
- после отметки мастера «Выполнено» заявка автоматически закрывается через 24 часа, если спор не открыт;
- при отмене мастером на раннем этапе поиск перезапускается с первой волны, а мастеру фиксируется санкция;
- при отсутствии мастеров HOLD аннулируется.

## 4. Плановый режим

Клиент создаёт опубликованную заявку с датой, временным слотом, районом, описанием и необязательным бюджетом.

Ограничения:

- горизонт планирования — 14 дней;
- максимум 5 ставок на заявку;
- один мастер может сделать только одну ставку;
- подтверждение выбранным мастером — в течение 2 часов;
- автозакрытие после выполнения — 24 часа.

В текущей реализации каждая ставка уменьшает баланс мастера на один lead-кредит и создаёт операцию `SPEND`.

Основная цепочка:

`CREATED → PUBLISHED → MASTER_SELECTED → CONFIRMED → IN_PROGRESS → DONE → CLOSED`.

Также предусмотрены `EXPIRED`, отмены обеими сторонами и `DISPUTE`.

## 5. Realtime

Клиент отправляет:

- `presence:online`;
- `presence:offline`;
- `geo:update`.

Сервер отправляет, в зависимости от сценария:

- `offer:new`;
- `offer:closed`;
- `order:status`;
- `master:location`;
- `bid:new`;
- `bid:selected`;
- `bid:closed`;
- события изменения плановой заявки и операторских действий.

`geo:update` мастера во время поездки пересылается клиенту вместе с расчётным ETA.

## 6. Фоновые задания

pg-boss используется для:

- последовательных волн срочного матчинга;
- истечения офферов;
- таймаута подтверждения цены;
- автозакрытия срочной заявки;
- таймаута подтверждения выбранным мастером;
- истечения плановой заявки;
- автозакрытия плановой заявки.

В e2e pg-boss можно отключить через `PGBOSS_DISABLED=1`.

## 7. Платежи в текущей реализации

Текущий `MockPaymentProvider` всегда возвращает успешный результат, но создаёт реальные записи бизнес-журнала:

- `HOLD`;
- `CAPTURE`;
- `VOID`;
- mock charge для покупки lead-кредитов;
- mock payout для вывода;
- mock refund.

Это означает, что интеграции с банком ещё нет, однако финансовая state machine уже действует. Mock-платежи нельзя считать бесплатным пилотным режимом.

## 8. Frontend

Фактический frontend-стек:

- React 19;
- Vite;
- TypeScript;
- React Router;
- TanStack Query;
- Tailwind CSS;
- Socket.IO client;
- Leaflet;
- i18next;
- PWA plugin.

Один frontend обслуживает три пользовательских контура:

- клиент;
- мастер;
- оператор.

Доступные экраны и действия зависят от JWT-пользователя, наличия профиля мастера, статуса верификации и роли оператора.

## 9. Локальный запуск

Инфраструктура разработки включает две PostGIS-базы:

- `localhost:5432` — development;
- `localhost:5433` — e2e.

Базовый порядок:

```bash
docker compose up -d
pnpm install
cp apps/api/.env.example apps/api/.env
cd apps/api
pnpm prisma migrate dev
pnpm prisma db seed
cd ../..
pnpm --filter api start:dev
pnpm --filter web dev
```

## 10. Что ещё не является production-ready

- реальный SMS-провайдер;
- реальный Kaspi/payment provider;
- объектное хранилище вместо локального диска;
- production deployment и оркестрация;
- централизованные логи, метрики и алерты;
- управление секретами;
- формализованные backup/restore;
- rate limits и усиленная antifraud-защита;
- согласованная политика хранения персональных данных;
- полноценная спецификация OpenAPI и каталог WebSocket-событий;
- выделенный бесплатный пилотный режим.
