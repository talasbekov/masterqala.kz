# Клиент v2: фронтенд строго по хендофф-прототипу — дизайн

> **Для агентов-исполнителей:** это дизайн-документ верхнего уровня. Он декомпозирует Цикл 2 на 4 под-цикла (Фазы A–D), каждый со своим отдельным spec→plan→SDD циклом (см. `superpowers:brainstorming` — «если проект слишком велик для одного плана, декомпозировать на под-проекты»). Не пытаться реализовать всё одним планом.

## Контекст и цель

Пользователь передал хай-фай кликабельный прототип `apps/MasterQala/design_handoff_masterqala/Этап 5 - Клиент (mobile).dc.html` (1051 строк, симуляция состояний на JS, single-file) и потребовал «строго разрабатывать под прототип». Бэкенд уже расширен под прототип (Цикл 1, PR #1: фото, детали адреса, сохранённые адреса, характеристики мастера в ставках, бюджет+слот, живая геопозиция+ETA, дедлайны-таймстемпы, приватность оффера, минимальный Rating). Цикл 2 — привести `apps/web` к визуальному и поведенческому соответствию прототипу для роли Клиент (mobile). Роли Мастер и Оператор — вне скоупа, отдельные будущие циклы.

Прототип определяет **31 экран-состояние** (`scr.splash` … `scr.payments`, подтверждено построчным grep). Ниже — ключевое архитектурное упрощение: большая часть этих 31 состояния — не отдельные URL-маршруты, а **проекции реального статуса заявки** (`Order.status`/`PlannedOrder.status`), которые уже существуют в бэкенде.

## Архитектурные решения

### 1. Маршруты вместо однофайловой state-machine

Прототип — один HTML-файл с JS-переключением `scr.*`. Реальный продукт не обязан копировать этот механизм 1:1 — только визуальный результат и переходы. React Router (уже используется в кодовой базе) с вложенными маршрутами — идиоматичнее и не требует ре-архитектуры существующего `App.tsx`.

**Итоговая карта маршрутов** (заменяет текущие client-маршруты, мастер/админ не трогаем):

| Маршрут | Экран(ы) прототипа | Примечание |
|---|---|---|
| `/login` | splash, phone, sms | 3 внутренних шага одного компонента (как сейчас, но добавить splash) |
| `/` | home | редизайн токенами v2 |
| `/catalog` | catalog | новый |
| `/order/new` | uw1, uw2, uw3, uw4 | 4-шаговый визард вместо одной формы |
| `/order/:id` | search, nomasters, track, price, progress, done, closed | **один маршрут, вид переключается по `order.status`**: SEARCHING→search, NO_MASTERS→nomasters, ACCEPTED/ON_WAY/ON_SITE→track, PRICE_PROPOSED→price, IN_PROGRESS→progress, COMPLETED→done, CLOSED→closed(+звёзды рейтинга) |
| `/order/:id/dispute` | dispute | форма/статус спора, как отдельный под-вид |
| `/planned/new` | pw1, pw2, pw3 | 3-шаговый визард |
| `/planned/:id` | pwait, compare, pconfirm, pactive, +progress/done/closed(общие с uw) | тот же принцип: вид по `plannedOrder.status` |
| `/planned/:id/dispute` | dispute | тот же компонент, что и для срочной |
| `/orders` | orders | история (объединённый список, как сейчас) |
| `/orders/:type/:id` или переиспользование `/order/:id`\|`/planned/:id` | detail | закрытая/прошлая заявка — тот же маршрут детали, просто статус CLOSED |
| `/support` | support | новый |
| `/notifications` | notifications | новый (пока без бэкенда пушей — статический список последних событий, реальные данные из уже существующих сущностей заявок/ставок, без отдельной таблицы уведомлений — уточнить объём в плане Фазы D) |
| `/profile` | profile | редизайн, + переключатель RU/KK/EN (см. §3) |
| `/profile/addresses` | addresses | CRUD, бэкенд уже есть (Cycle 1, Task 6) |
| `/profile/payments` | payments | **статическая заглушка «скоро»** (решение пользователя) |

Итого ~15 реальных маршрутов вместо 31 «экрана» — управляемо.

### 2. Токены v2 — скоуп только на клиента

Пользователь ранее выбрал «Токены v2 → Клиент» — НЕ переписывать глобальные токены `packages/ui` (это затронуло бы мастер/админ страницы и Claude Design sync, см. [[masterqala-stage-plan]]). Вместо этого:

- Новый файл `packages/ui/src/tokens-client-v2.css`, все переменные — **внутри скоуп-класса** `.qala-client-v2` (не в `:root`/`@theme` глобально).
- Значения (подтверждено из прототипа, строки 13, 43, 49 и по всему файлу): primary/accent `#166088` (hover `#134F73`), page bg `#C7DAE3`, mobile inner bg `#F2F7FA`, secondary surface/border `#C0D6DF`, light fills `#CFE0E8`/`#EAF2F6`/`#F6FAFC`, text `#14303C`, secondary text `#5B7B8A`, muted `#8FAAB8`, text-on-fill `#38596A`, success `#059669`/bg `#E5F3EE`/text `#065F46`, danger `#DC2626`/bg `#FBECEC`/text `#991B1B`, warning bg `#FFF1E8`/text `#B4530A`. Шрифт Manrope (уже есть в пакете с этапа design-sync).
- «Page bg» (`#C7DAE3`) и рамка/скругление 44px в прототипе — это хром инструмента прототипирования (имитация корпуса телефона на десктопе), НЕ часть реального продукта. Реальный PWA рендерится в настоящем мобильном браузере — используем `#F2F7FA` как фон приложения напрямую, без фейкового «корпуса телефона». Это решение архитектурное, не расхождение с прототипом по существу.

### 3. Новый слой компонентов, не `packages/ui`

Компоненты этого цикла (степпер визарда, bottom sheet трекинга, звёздный рейтинг, таб-бар) специфичны для мобильного клиентского флоу и не обобщаются на другие роли. Живут в `apps/web/src/features/client-v2/components/`. Существующие `@masterqala/ui` примитивы (`Avatar`, иконки) переиспользуются там, где подходят как есть; `Button`/`Card` v1 — НЕ переиспользуются (другая визуальная система), новые локальные аналоги под токены v2.

Существующие redesigned-v1 страницы (`HomePage`, `NewOrderPage`, `OrderPage`, `MyOrdersPage`, `PlannedNewOrderPage`, `PlannedOrderPage`) и старые нередизайненные (`ProfilePage`) — **полностью заменяются**, не патчатся инкрементально.

### 4. Карта — react-leaflet + OSM

Пользователь подтвердил «реальная карта, соответствует хендоффу» ранее в сессии. Прототип буквально использует OSM-тайлы (`tile.openstreetmap.org`, строки 312-315). Библиотеки: `leaflet` + `react-leaflet` (без API-ключа, OSM tile layer). Используется в `track`-виде `/order/:id` — позиция мастера уже транслируется бэкендом (Cycle 1: живая геопозиция + ETA через `geo:update` socket-событие и `MasterPresence.location`).

### 5. i18n — только инфраструктура + RU-контент

Пользователь подтвердил «только инфраструктура + RU-контент» ранее. Прототип на экране `profile` (строки 770-772) показывает переключатель Рус/Қаз/Eng — UI переключателя нужен (три кнопки, активное состояние), но переводит контент **только RU**; KK/EN — файлы-заглушки (`kk.json`/`en.json` = копия ru.json структуры или маркер untranslated), переключение реально меняет только визуальное состояние кнопки, не тексты (либо честно: fallback на RU при отсутствии перевода — так и работает `react-i18next` из коробки). Библиотека: `react-i18next` + `i18next` (индустриальный стандарт, легко расширить позже).

### 6. Рейтинг — уже готовый бэкенд

Экран `closed` (строка 410-422) показывает 5 звёзд «Оцените Данияра» сразу на экране успеха. Бэкенд уже отдаёт `POST /orders/:id/review` / `POST /planned-orders/:id/review` (коммит `cd9a3f7`, этот же цикл сессии) — Фаза B/C реализует только UI компонент звёзд + вызов уже существующего эндпоинта.

### 7. Способы оплаты — статическая заглушка

Экран `payments` — решение пользователя «скоро», без реального управления картами Kaspi. Простой экран с иконкой/текстом, доступный из Profile, без форм/API-вызовов.

## Декомпозиция на фазы (каждая — свой spec→plan→SDD цикл)

**Фаза A — Фундамент: тема v2, shell, авторизация, главная, каталог.** Разблокирует все остальные фазы.
- `tokens-client-v2.css` + Tailwind wiring в `apps/web`
- Mobile app shell (контейнер, нижний таб-бар Главная/Заявки/Профиль)
- i18n-скаффолд (`react-i18next`, ru-namespace для строк фазы A, kk/en-заглушки)
- `LoginPage` v2 (splash→phone→sms)
- `HomePage` v2
- `CatalogPage` v2 (новый)

**Фаза B — Срочный режим.** Самая сложная: визард + живая карта + рейтинг.
- Визард `/order/new` (uw1–uw4) взамен `NewOrderPage`
- `/order/:id`: search, nomasters, track (+Leaflet+ETA), price, progress, done, closed(+звёзды рейтинга через уже готовый API)
- `/order/:id/dispute`

**Фаза C — Плановый режим.**
- Визард `/planned/new` (pw1–pw3) взамен `PlannedNewOrderPage`
- `/planned/:id`: pwait, compare (ставки с рейтингом мастера — уже в API из Rating-фичи), pconfirm, pactive + общие progress/done/closed
- `/planned/:id/dispute`

**Фаза D — История, профиль, вспомогательные экраны.**
- `/orders` v2 (взамен `MyOrdersPage`)
- `/profile` v2 (взамен старой нередизайненной ProfilePage) + переключатель языка
- `/profile/addresses` (CRUD, бэкенд готов)
- `/profile/payments` (заглушка)
- `/support`
- `/notifications`

## Явно вне скоупа

- Роли Мастер (mobile) и Оператор (desktop) — отдельные будущие циклы, прототипы для них существуют (`Этап 5 - Мастер (mobile).dc.html`, `Этап 6 - Оператор (desktop).dc.html`), не читались в этом цикле.
- Десктопная адаптация клиента — мобильный экран, как в прототипе.
- Реальное хранение карт Kaspi — заглушка.
- Push-уведомления как отдельная бэкенд-сущность — экран `notifications` рендерит доступные данные (заявки/ставки), не заводим новую таблицу уведомлений без явного запроса.
- Автоматизированные фронтенд-тесты — в репозитории их нет и не заводились для прошлого редизайна (v1); верификация — `pnpm --filter web build` + живая проверка в браузере, как и раньше.

## Самопроверка

Все 31 состояние прототипа (построчный grep `sc-if value="{{ scr\.`) учтены в карте маршрутов выше. Токены v2 подтверждены прямым чтением CSS блока прототипа (строка 13) и повторяющихся hex-значений по всему файлу. Архитектурные решения (скоуп темы, react-leaflet, i18n библиотека) соответствуют ранее данным пользователем ответам через AskUserQuestion в этой же сессии (см. память [[masterqala-stage-plan]]). Плейсхолдеров/TBD нет — каждое из 4 фаз имеет чёткий список экранов и маршрутов.
