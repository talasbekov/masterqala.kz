# Дизайн: визуальная система клиентского потока (MasterQala.kz)

Дата: 2026-07-17
Статус: утверждено пользователем в брейнсторминге, готово к плану реализации.

## Контекст

MVP бэкенда полностью реализован (5 этапов, ветка `stage1-foundation`). Фронтенд (21 страница, `apps/web`) написан функционально по ходу реализации бэкенда, без дизайн-системы — один цвет (`teal-700`), пустой `index.css` (только `@import "tailwindcss"`). Задача — редизайн визуального слоя поверх существующей бизнес-логики: контракты данных, хуки, роутинг не меняются.

Скоуп этой итерации — клиентский поток (7 страниц), не весь фронтенд. Роль мастера и админка оператора — вне скоупа.

## Продукт и платформа

Маркетплейс бытовых услуг «на дом» (Казахстан): клиент вызывает мастера (сантехник/электрик/уборка/ремонт техники/кондиционеры/мебель) через мобильное PWA. Два режима: срочный (система сама находит мастера, live-статусы по WebSocket) и плановый (публикация заявки, ставки мастеров, выбор).

Mobile-first PWA, десктоп не нужен — только узкий экран (~375–430px). Стек: React 19 + Vite + Tailwind CSS 4, React Router, TanStack Query, socket.io-client.

## Визуальное направление

Референс по ощущению — приложения вроде Яндекса: тёплая, уютная, умеренно плотная (не пустая, не перегруженная) подача, с первого взгляда понятно, что это сервис вызова мастеров. Утверждено после итерации в визуальном компаньоне (первая версия — блёкло-голубая, минималистичная — была отклонена как «непонятная и слабая»; вторая версия с сеткой категорий и тёплым фоном — принята за основу).

Источник цветовой палитры и шрифтовой пары — скилл `ui-ux-pro-max`, домен продукта «Home Services (Plumber/Electrician)» — наиболее точное совпадение по типу продукта в его базе данных.

### Токены

Tailwind 4 — CSS-first конфиг через `@theme` в `apps/web/src/index.css` (не `tailwind.config.js`):

- `--color-primary: #1E40AF`, `--color-primary-light: #3B82F6` — синий, логика «спокойствие/доверие»: вторичные действия, статусные бейджи
- `--color-accent: #EA580C` — оранжевый, логика «срочное/основное действие»: кнопка «Вызвать мастера», ключевые CTA
- `--color-background: #FFFCF5` — тёплый кремовый фон экрана (не холодный/стерильный)
- `--color-surface: #FFFFFF` — фон карточек
- `--color-foreground: #1B1B1F`, `--color-muted: #8A8A8F`, `--color-border: #F0EAE0`
- `--color-success: #059669`, `--color-destructive: #DC2626`, `--color-rating: #F59E0B` (★)
- `--radius-sm: 10px`, `--radius-md: 14px`, `--radius-lg: 18px`, pill = `999px`
- Мягкие тени вместо плоских блоков без теней
- Шрифт — Manrope (Google Fonts), гуманистический sans-serif

## Компоненты (`apps/web/src/components/ui/`)

Строим только то, что нужно семи страницам клиентского потока — без задела на будущее (YAGNI):

| Компонент | Назначение |
|---|---|
| `Button` | primary (оранжевая заливка, срочные действия) / secondary (синий контур), full-width на мобильном |
| `Card` | базовый контейнер: `--radius-md`, мягкая тень, `--color-surface` |
| `CategoryTile` | иконка в цветном скруглённом квадрате + подпись — сетка категорий на HomePage |
| `Avatar` | инициалы или фото мастера, круг |
| `RatingBadge` | ★ + число |
| `StatusPill` | цветная плашка статуса заявки — маппинг цвета на семантику (поиск/в пути = синий или оранжевый в зависимости от срочности, выполнено = зелёный, отменено/спор = красный), не единый цвет на все статусы |
| `BottomTabBar` | замена `components/TabBar.tsx`: иконки + подписи, активное состояние |
| `EmptyState` | пустой список заявок на `MyOrdersPage` |
| Иконки | inline SVG, monoline stroke — поиск, гаечный ключ (сантехник), молния (электрик), галочка (уборка), шеврон, дом, список, профиль. Без эмодзи (правило скилла `ui-ux-pro-max`) |

Расположение: сначала внутри `apps/web/src/components/ui/`, без отдельного pnpm-пакета — быстрее итерировать, пока API компонентов не устоялся на реальных экранах. Вынос в `packages/ui` как отдельный воркспейс-пакет — отдельная механическая задача перед первым реальным прогоном `/design-sync` (design-sync ожидает собираемый npm-пакет компонентов, а не под-папку внутри app).

## Экраны и порядок применения

В порядке пользовательского пути, от простого к сложному:

1. `LoginPage` — SMS-логин
2. `HomePage` — точка входа: активная заявка (карточка с аватаром/рейтингом/ETA), сетка категорий, CTA «Вызвать мастера», строка «Запланировать»
3. `NewOrderPage` — создание срочной заявки
4. `OrderPage` — отслеживание срочной заявки (live-статусы через `order:status` сокет)
5. `MyOrdersPage` — список заявок клиента (+ `EmptyState`)
6. `PlannedNewOrderPage` — публикация плановой заявки
7. `PlannedOrderPage` — плановая заявка: ставки мастеров (`bids[]`), выбор (`selectedBidId`)

## Данные (контракт не меняется)

- `Order`: `category.name`, `status` (enum), `address`, `master{name, phone — раскрывается только после CONFIRMED}`, цена/`calloutPrice`, `createdAt`
- Статусы срочной заявки — живые, обновляются через WebSocket `order:status`
- `PlannedOrder`: `bids[]` от мастеров с ценой, `selectedBidId`
- Рейтинг мастера — если поле есть в ответе API; если нет, `RatingBadge` скрывается (не выдумывать данные)

## Тестирование

Только визуальный слой — хуки, API-вызовы, роутинг не трогаются. Проверено: все e2e-тесты (`apps/api/test/*.e2e-spec.ts`) — бэкендовые, ни один не завязан на вёрстку `apps/web`. Редизайн не ломает существующее тестовое покрытие.
