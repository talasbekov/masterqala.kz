# Клиент v2 — Фаза B: срочный режим — дизайн

> Продолжение Цикла 2 (см. `docs/superpowers/specs/2026-07-19-client-v2-frontend-design.md`). Большинство архитектурных решений уже приняты и подтверждены в Фазе A (токены `c2`, i18n, `apps/web/src/features/client-v2/` как новый слой, статус-driven единый маршрут вместо отдельных URL на каждое состояние прототипа) — здесь фиксируются только решения, специфичные для срочного режима.

## Скоуп

Экраны прототипа `uw1-uw4, search, nomasters, track, price, progress, done, closed` + `dispute` (для срочной заявки). Заменяют `apps/web/src/pages/NewOrderPage.tsx` и `apps/web/src/pages/OrderPage.tsx` полностью (не патчатся).

## Маршруты

| Маршрут | Экраны прототипа | Примечание |
|---|---|---|
| `/order/new` | uw1, uw2, uw3, uw4 | 4-шаговый визард, один компонент, внутреннее состояние шага |
| `/order/:id` | search, nomasters, track, price, progress, done, closed | вид переключается по `order.status` (см. карту ниже) |
| `/order/:id/dispute` | dispute | отдельный маршрут (не модалка — прототип показывает полноэкранную форму) |

Карта `order.status` → вид:
- `SEARCHING` → search
- `NO_MASTERS` → nomasters
- `ACCEPTED` / `MASTER_ON_WAY` / `INSPECTION` → track
- `AWAITING_PRICE_CONFIRM` → price
- `IN_PROGRESS` → progress
- `DONE` → done
- `CLOSED` / `CANCELLED_BY_CLIENT` / `CANCELLED_BY_MASTER` → closed (для отмен — упрощённый вид без звёзд рейтинга, см. ниже)

## Ключевые решения

### 1. Карта — `react-leaflet` + OSM, один переиспользуемый компонент

Прототип использует статичные OSM-тайлы (4 картинки внахлёст) на трёх экранах: uw3 (выбор адреса), search (пульсирующая точка своей геопозиции), track (живая позиция мастера). Вместо копирования статичных тайлов — один реальный интерактивный компонент `apps/web/src/features/client-v2/components/MapView.tsx` на `react-leaflet`+`leaflet` (OSM tile layer, без API-ключа), с тремя режимами через пропсы:
- `pin` (uw3): перетаскиваемый маркер по центру, callback с новыми координатами при движении карты
- `pulse` (search): статичная точка с CSS-анимацией пульсации (не Leaflet-маркер — точнее повторяет прототип)
- `tracking` (track): маркер мастера, обновляемый через `master:location` сокет-событие, + фиксированный маркер точки назначения (адрес заявки)

Библиотеки: `leaflet`, `react-leaflet`, `@types/leaflet` — добавляются в `apps/web/package.json`.

### 2. Визард — 4 шага, состояние в одном компоненте

`apps/web/src/features/client-v2/pages/NewOrderPage.tsx`, `useState<1|2|3|4>` для шага, накопленное состояние заявки в одном объекте. Отправка на бэкенд — один вызов `POST /orders` на шаге 4 (не пошагово).

- **Шаг 1** (uw1): сетка категорий 2 колонки — переиспользует `categoryMeta()` из Фазы A.
- **Шаг 2** (uw2): текстовое описание + фото (до 5, реальная загрузка через существующий `POST /uploads` → `apiUpload()`, накопление `photoPaths: string[]`).
- **Шаг 3** (uw3): `MapView` в режиме `pin` для выбора координат + быстрый выбор сохранённого адреса. **Сохранённые адреса бэкенд уже поддерживает (`GET/POST/PATCH/DELETE /addresses`, Cycle 1), но фронтенд их ещё нигде не использует** — Фаза B первой их подключает: чипы «Дом»/«Работа»/сохранённые + «＋ новый». Поля подъезд/этаж/квартира/комментарий — прямые инпуты (не выбор из адреса, т.к. это специфика конкретной заявки, не самого адреса). `district` — обязательное поле бэкенда (Cycle 1) — заполняется автоматически через обратное геокодирование не делаем (нет сервиса геокодирования в проекте); вместо этого — простой текстовый инпут «Район» рядом с адресом, заполняемый вручную (минимальное расхождение с прототипом, который не показывает отдельное поле района на этом экране явно, но того требует бэкенд-контракт).
- **Шаг 4** (uw4): сводка + `POST /orders/preview` для расчёта (`calloutPrice`/`serviceFee`) + статичная строка способа оплаты (заглушка «Kaspi Gold ···· 4821, сменить» — некликабельно ведёт на `/profile/payments`-заглушку из будущей Фазы D; сейчас просто статичный текст без перехода, т.к. `/profile/payments` ещё не существует) + кнопка «Найти мастера» → `POST /orders`.

### 3. Order detail — общий каркас + виды по статусу

`apps/web/src/features/client-v2/pages/OrderPage.tsx` — грузит заявку (`GET /orders/:id` + `order:status` сокет, как в v1), рендерит один из 7 видов по статусу (отдельные подкомпоненты в `apps/web/src/features/client-v2/components/order-views/`). Общая логика загрузки/сокета — в самом `OrderPage`, виды — чистые презентационные компоненты, принимающие `order` и колбэки действий.

- **search**: `MapView` в режиме `pulse` + текст волны (локально по `order.wave`: 0→«Начинаем поиск…», 1→«Ищем мастера в радиусе 3 км…», 2→«Расширяем поиск до 6 км…», 3→«Расширяем поиск до 10 км…» — те же значения, что уже есть в `apps/web/src/orderStatus.ts` `WAVE_TEXTS`, переиспользуются) + время с момента `order.createdAt` (локальный таймер, т.к. бэкенд не отдаёт отдельный `searchDeadline`) + «Отменить — бесплатно» → `POST /orders/:id/cancel`.
- **nomasters**: «Никто не откликнулся» + «Повторить поиск» (`POST /orders/:id/retry-search`) + «Запланировать на время» (`navigate('/planned/new')`) + «На главную».
- **track**: `MapView` в режиме `tracking` (слушает `master:location`: `{orderId, lat, lng, etaMinutes}`) + карточка мастера (имя, «✓ проверен» если `master.verified`, рейтинг `★ {master.rating} · {master.reviewCount} заказов` — поля уже приходят из `attachRating`, Cycle-1-эпилог) + кнопка звонка (`<a href="tel:{master.phone}">`) + ETA-плашка + «Поддержка»/«Отменить» (после принятия — по прототипу отмена после принятия платная, тот же `POST /orders/:id/cancel`, бэкенд сам решает удержание).
- **price**: разбивка (выезд/работы/итого из `calloutPrice`/`workPrice`), обратный отсчёт от `priceDeadline` (уже вычисляется бэкендом), «Подтвердить»/«Отклонить цену» → `confirm-price`/`reject-price`.
- **progress**: степпер (переиспользать существующий `STEPPER_STEPS` из `orderStatus.ts`, уже есть) + «Поддержка».
- **done**: сводка суммы + «Подтвердить выполнение» (`confirm-completion`) + «Открыть спор» → `navigate('/order/:id/dispute')`.
- **closed**: для настоящего `CLOSED` (не отмены) — блок звёзд рейтинга, если `!order.review`: `POST /orders/:id/review` (уже готовый API, Rating-фича этой же сессии) с `{rating, comment?}`; если `order.review` уже есть — показать «Спасибо за отзыв» вместо формы. Для `CANCELLED_BY_CLIENT`/`CANCELLED_BY_MASTER` — тот же контейнер, но без звёзд (отменённую заявку не оценивают), с `cancelReason`, если есть.

### 4. Dispute — полноэкранная страница, богаче текущего `DisputeCard`

Текущий `DisputeCard` в `OrderPage.tsx` (v1, инлайн-компонент, не отдельный файл) — минимальный (textarea + upload). Прототип показывает более богатый экран: чипы категории (Качество работ/Не та цена/Поведение/Другое) + textarea + до нескольких фото + после отправки — статус-таймлайн. Бэкенд принимает только свободный текст `reason` (`OpenDisputeDto.reason: string`, без категорий на бэке) — чипы реализуются на фронте как удобство: выбранная категория подставляется префиксом в итоговый `reason` (`"Категория. <текст из textarea>"`), без изменений бэкенда. Новый файл `apps/web/src/features/client-v2/pages/DisputePage.tsx` на маршруте `/order/:id/dispute`, использует существующие `POST /orders/:id/disputes`, `POST /disputes/:id/evidence`, `PATCH /disputes/:id`. Старый инлайн `DisputeCard` в v1 `OrderPage.tsx` не трогаем (v1 файл целиком заменяется/удаляется вместе с остальным).

### 5. Что не переносим буквально из прототипа (архитектурные, не содержательные отклонения)

- OSM-тайлы как статичные `<img>` → реальный Leaflet (уже решено в Фазе A design spec — «Да, реальная карта»).
- Таймеры (searchTimeStr, priceTimerStr) — не хардкод-демо, а реальные вычисления от реальных полей заявки.
- Способ оплаты на шаге 4 визарда — статичная строка без реального переключения карт (совпадает с уже принятым решением «заглушка «скоро»» для полноценного экрана способов оплаты, Фаза D).

## Самопроверка

Все 4 экрана визарда, все 7 статус-видов и dispute-экран учтены. Контракты сверены напрямую с кодом (`orders.controller.ts`, `orders.service.ts`, `order.constants.ts`, `dto.ts`, `disputes/dto.ts`, `realtime.gateway.ts`) — не с памятью/резюме. Единственное реальное расхождение с прототипом по существу — обязательное поле «район», которого нет explicit на экране uw3 прототипа, но требует бэкенд; решение (простой текстовый инпут) явно зафиксировано, не оставлено как TBD.
