# Этап 2 «Срочный режим» — технический дизайн

> Основание: бизнес-спека `docs/project-spec.md` (§3.3 срочная заявка, §3.5 матчинг, §3.6 ценообразование, §3.8 завершение, §3.9 отмена, §5.1 state machine, §6 числа). Все бизнес-решения там закрыты; этот документ фиксирует техническую реализацию поверх фундамента этапа 1.

## 1. Скоуп

**Входит:** полный жизненный цикл срочной заявки — создание с превью цены, матчинг волнами, принятие, статусы до закрытия, подтверждение цены работ, отмены обеих сторон, таймауты (волны 60/90с, цена 15 мин, авто-закрытие 24ч), «нет мастеров → повторить»; presence и живая геолокация мастеров; realtime-обновления; оплата сервисного сбора через mock-провайдер; начисления компенсаций мастеру (записи, без кошелька); экраны клиента и мастера; нижняя таб-навигация.

**Не входит:** плановый режим и lead-кредиты (этап 3), реальный Kaspi и кошельки/вывод (этап 4), споры и штрафы кредитами (этап 5; отмена мастером в этапе 2 фиксируется в журнале, санкции применит этап 5), Web Push (мастер держит PWA открытой; отдельный заход позже), геокодинг и реальный routing-провайдер (внешние зависимости — адаптеры готовы), рейтинги (фаза 2).

## 2. Архитектурные решения (приняты 2026-07-14/15)

| Область | Решение |
|---|---|
| Realtime | WebSocket: Nest gateway + socket.io; авторизация JWT в handshake; presence мастеров через heartbeat |
| Таймеры | pg-boss (очередь в той же PostgreSQL, без Redis): отложенные джобы для волн и таймаутов |
| Ядро | Событийный оркестратор: атомарные гейты `updateMany({id, status: from})` (паттерн этапа 1) + джобы; WS-события после коммита |
| Гео мастера | Живая браузерная геолокация при «онлайн», периодическая отправка → PostGIS point |
| Расстояние | DI-токен `ROUTING_SERVICE`; dev-реализация: PostGIS `ST_Distance` × 1.3; реальный 2ГИС/Google — позже |
| Оплата | DI-токен `PAYMENT_PROVIDER`: `hold/capture/void`; `MockPaymentProvider` всегда успешен, пишет `PaymentTransaction` |
| Адрес клиента | Браузерная геолокация (с подтверждением) + адрес текстовым полем; без геокодинг-API |
| Лимит | У клиента одна активная срочная заявка одновременно (409 при попытке второй) |

## 3. Модель данных (Prisma + raw SQL для PostGIS)

- **Order**: id, clientId→User, categoryId→Category, description, address (текст), `location geography(Point,4326)` (raw-миграция + raw-запросы), status `OrderStatus`, masterId→User?, wave (0–3), `calloutPrice` (₸, int), `serviceFee` (₸, int), `workPrice` (₸, int?), причина отмены?, таймстемпы переходов (acceptedAt, onSiteAt, priceProposedAt, completedAt, closedAt), createdAt/updatedAt.
- **OrderStatus** (enum, UI по-русски): `CREATED, SEARCHING, ACCEPTED, MASTER_ON_WAY, INSPECTION, AWAITING_PRICE_CONFIRM, IN_PROGRESS, DONE, CLOSED, NO_MASTERS, CANCELLED_BY_CLIENT, CANCELLED_BY_MASTER, DISPUTE` (DISPUTE заложен в enum, переходы — этап 5).
- **OrderOffer**: orderId, masterUserId, wave, sentAt, respondedAt?, outcome `OfferOutcome {PENDING, ACCEPTED, LOST, EXPIRED}` — журнал рассылки; уникальность (orderId, masterUserId) — мастер не получает один заказ дважды.
- **MasterPresence**: masterUserId (unique), isOnline, lastSeenAt, `location geography(Point,4326)?`. Оффлайн-джоба: lastSeenAt старше 2 мин → isOnline=false.
- **PaymentTransaction**: orderId, type `{HOLD, CAPTURE, VOID}`, amount, status `{PENDING, SUCCEEDED, FAILED}`, providerRef, createdAt.
- **Accrual** (начисления мастеру, без кошелька): masterUserId, orderId, type `{CALLOUT_COMPENSATION}`, amount, createdAt. Этап 4 превратит записи в денежный баланс.
- pg-boss создаёт схему `pgboss` автоматически.

## 4. State machine (`OrdersService`)

Переходы — §5.1 бизнес-спеки один-в-один. Каждый переход: транзакция → `updateMany({id, status: from})`, `count===0` → 409 «Заявка в другом статусе» → побочные эффекты (платёж/начисление/джоба) → после коммита WS-события. Кто может дергать переход — гард по роли и принадлежности (клиент заявки / назначенный мастер).

Ключевые связки:
- `CREATED→SEARCHING`: hold сервисного сбора (mock) → джоба `wave(orderId, 1)`.
- `SEARCHING→ACCEPTED` («Принять» мастером): гейт + проверка «мастер свободен» (нет активной заявки) в той же транзакции; capture холда; остальным офферам волны — `LOST` + WS «заявка принята другим».
- `AWAITING_PRICE_CONFIRM`: джоба таймаута 15 мин → авто-`CANCELLED_BY_CLIENT` (с пометкой «таймаут»), начисление компенсации мастеру.
- `DONE→CLOSED`: подтверждение клиентом или джоба 24ч (авто); при закрытии — начисление `CALLOUT_COMPENSATION` = calloutPrice − serviceFee.
- Отмены (§3.9): клиент до ACCEPTED — void холда, бесплатно; клиент после ACCEPTED (включая отклонение цены) — сбор удержан (capture уже был), компенсация мастеру начисляется; мастер после ACCEPTED — `CANCELLED_BY_MASTER` → автоматический перезапуск поиска (заявка возвращается в SEARCHING с волны 1, исключая отменившего мастера), факт в OrderOffer-журнале.
- `NO_MASTERS`: void холда; кнопка «Повторить поиск» → снова SEARCHING (новый hold).

## 5. Матчинг (`MatchingService` + pg-boss)

Джоба `wave(orderId, n)`:
1. Заявка не в SEARCHING → выход (идемпотентность).
2. Кандидаты (raw SQL): профиль ACTIVE, presence isOnline, `ST_DWithin(location, order.location, радиус волны)` (3/6/10 км), категория совпадает, нет активной заявки, нет OrderOffer по этой заявке.
3. Есть кандидаты → создать OrderOffer(PENDING) каждому + WS-оффер (карточка + компенсация + дедлайн волны).
4. Поставить `wave-timeout(orderId, n)` через 60с (волны 1–2) / 90с (волна 3).

Джоба `wave-timeout(orderId, n)`: заявка всё ещё SEARCHING → PENDING-офферы волны → EXPIRED; n<3 → `wave(orderId, n+1)`, иначе → `NO_MASTERS`.

Гонка «Принять»: атомарный гейт SEARCHING→ACCEPTED решает; проигравшие получают 409 + WS.

## 6. Ценообразование (`PricingService`)

`calloutPrice = round((2000 + км × 150) × коэф_времени)`; коэф.: 08–20 ×1.0, 20–23 ×1.2, 23–08 ×1.5 (время Алматы); `serviceFee = max(round(calloutPrice × 0.40), 1000)`. Км — `RoutingService.distanceKm(from, to)`; dev: `ST_Distance` (по прямой, метры) × 1.3. Превью до создания: `POST /orders/preview {categoryId, lat, lng}` → расстояние до ближайшего подходящего онлайн-мастера; нет мастеров в 10 км → превью с флагом «мастеров рядом нет». Тарифы — константы конфиг-модуля (env-переопределение), в БД не выносим (YAGNI).

## 7. Realtime (`RealtimeGateway`)

- Handshake: JWT в auth → userId/роль; комнаты `user:{userId}`, при открытой заявке `order:{orderId}`.
- Мастер → сервер: `presence:online {lat,lng}`, `presence:offline`, `geo:update {lat,lng}` (~каждые 30с), heartbeat встроенный socket.io ping.
- Сервер → мастер: `offer:new {order-карточка, компенсация, дедлайн}`, `offer:closed {orderId, причина}`.
- Сервер → клиент: `order:status {status, master?, workPrice?}` на каждый переход.
- Fallback: клиент/мастер при reconnect и на входе рефетчат `GET /orders/active`.

## 8. HTTP API (основное)

- `POST /orders/preview` — превью цены; `POST /orders` — создать (409 если активная есть); `GET /orders/active`, `GET /orders/:id`, `GET /orders` (история клиента).
- Переходы: `POST /orders/:id/accept` (мастер), `/on-way`, `/on-site`, `/propose-price {amount, comment?}`, `/confirm-price`, `/reject-price`, `/complete` (мастер), `/confirm-completion` (клиент), `/cancel` (обе стороны, серверная логика по таблице §3.9), `/retry-search` (клиент из NO_MASTERS).
- Мастер: `GET /master/active-order`; presence — только по WS.

## 9. Экраны (все — mobile-first, тексты по-русски)

**Навигация:** нижние табы «Главная / Мои заявки / Работа / Профиль». «Работа» видна только мастерам со статусом ACTIVE. «Панель оператора» остаётся ссылкой в Профиле.

**Клиент:**
- Главная: большая кнопка «Вызвать мастера»; при активной заявке вместо неё — карточка-ссылка на текущую заявку.
- Форма заявки (одна страница): чипы категорий → описание (textarea) → гео-блок (автоопределение + кнопка «Обновить», текстовое поле адреса) → живое превью «Выезд: N ₸» (пересчёт при смене категории) + примечание «Работа оплачивается мастеру напрямую после согласования цены» → кнопка «Вызвать мастера».
- Поиск: анимированный индикатор, честный текст «Ищем в радиусе 3 км… / Расширяем до 6 км…», общий таймер, «Отменить» (бесплатно).
- Активная заявка: карточка мастера (имя, телефон-ссылка — виден после принятия, категория) + вертикальный степпер (Принята → Едет → На месте → Осмотр → Цена → В работе → Выполнено), текущий шаг подсвечен, WS-обновления; контекстное действие внизу фиксированной кнопкой (Подтвердить цену N ₸ / Отклонить — с таймером 15 мин; Подтвердить выполнение; Отменить с показом последствий).
- Нет мастеров: «Никто не откликнулся» + «Повторить поиск» (перейти в плановый — этап 3, пока скрыто).
- Мои заявки: список с русскими статусами и датами.

**Мастер («Работа»):**
- Тумблер «Онлайн» (запрашивает геолокацию; при отказе — объяснение, что без гео офферы не приходят), индикатор соединения.
- Входящий оффер: полноэкранная карточка со звуком и вибрацией — категория, расстояние, описание, «Компенсация выезда: N ₸», крупная кнопка «Принять» с убывающим таймером волны; опоздал/проиграл — карточка закрывается с пояснением.
- Активная заявка: адрес + описание + телефон клиента; кнопки по статусу: «Еду» → «На месте» → форма «Стоимость работ» (сумма + комментарий) → «Ожидание подтверждения цены…» → «Выполнено»; «Отменить» с предупреждением.

## 10. Обработка ошибок и крайние случаи

Пустая волна (ни одного кандидата) → немедленно следующая волна без ожидания таймаута; таймаут ставится только когда офферы реально отправлены. Мастер ушёл оффлайн после получения оффера → «Принять» решает гейт по БД; обрыв WS у клиента → рефетч активной заявки при reconnect; двойные клики → идемпотентные гейты (409); рестарт API в середине волны → pg-boss доставит `wave-timeout` после подъёма; часовой пояс коэффициента — Asia/Almaty фиксированно.

## 11. Тестирование

E2e (supertest + test-БД): happy path целиком с прямым вызовом джоб-хендлеров (без реальных ожиданий); гонка «двое принимают одновременно» (1×200, 1×409); волны (пустая волна 1 → волна 2; все пустые → NO_MASTERS); таймауты цены и авто-закрытия (вызов хендлера); отмены по всем строкам таблицы §3.9 с проверкой PaymentTransaction/Accrual; лимит «одна активная»; PricingService — юнит-тесты формулы и коэффициентов времени. WS — socket.io-client: получение оффера, order:status, presence→isOnline. Web — build + ручной сквозной сценарий двумя окнами (клиент + мастер).
