# Клиент v2 — Фаза C: плановый режим — дизайн

> Продолжение Цикла 2. Архитектурные решения (токены `c2`, i18n, статус-driven единый маршрут вместо URL на каждое состояние прототипа, `apps/web/src/features/client-v2/`) — те же, что в Фазах A/B, здесь фиксируются только специфичные для планового режима.

## Скоуп

Экраны прототипа `pw1-pw3, pwait, compare, pconfirm, pactive` + `dispute` (для плановой заявки). Заменяют `apps/web/src/pages/PlannedNewOrderPage.tsx` и `apps/web/src/pages/PlannedOrderPage.tsx` полностью.

**Побочно найденный и исправляемый баг старого кода:** текущий `PlannedNewOrderPage.tsx` отправляет `scheduledAt`, которого нет в `CreatePlannedOrderDto` — бэкенд ожидает `slotStart`/`slotEnd` (добавлены в Cycle 1). Публикация плановой заявки через старый UI сейчас физически не проходит валидацию. Это не регресс новой реализации — старый код уже сломан, просто раньше это было незаметно, т.к. никто не проверял этот путь после Cycle 1.

## Маршруты

| Маршрут | Экраны прототипа | Примечание |
|---|---|---|
| `/planned/new` | pw1, pw2, pw3 | 3-шаговый визард |
| `/planned/:id` | pwait, pactive, done, closed | вид по `plannedOrder.status` |
| `/planned/:id/compare` | compare | отдельный подмаршрут (симметрично `/order/:id/dispute` из Фазы B) |
| `/planned/:id/dispute` | dispute | **тот же компонент `DisputePage`, что в Фазе B** — рефакторится на приём `kind: 'orders' | 'planned-orders'` (различие только в префиксе API-пути, форма/логика идентичны) |

`pconfirm` — **не отдельный маршрут**, а локальное состояние внутри `/planned/:id` (выбранный `bidId` перед вызовом `select()`), аналогично тому, как шаг 4 визарда в Фазе B не был отдельным URL.

Карта `plannedOrder.status` → вид:
- `PUBLISHED` → pwait (список ставок + переход на pconfirm/compare)
- `MASTER_SELECTED` → pactive (подраздел «ждём подтверждения»)
- `CONFIRMED` / `IN_PROGRESS` → pactive (подраздел «подтверждено», контакты открыты)
- `DONE` → done (сводка + подтвердить выполнение + спор)
- `CLOSED` → closed (звёзды рейтинга через уже готовый `POST /planned-orders/:id/review`)
- `CANCELLED_BY_CLIENT` / `CANCELLED_BY_MASTER` / `EXPIRED` → closed (без звёзд, причина/пояснение об истечении)

`CREATED` не встречается в реальном UI — `create()` сразу переводит в `PUBLISHED`.

## Ключевые решения

### 1. Визард — 3 шага вместо 4 (нет геолокации/карты)

`apps/web/src/features/client-v2/pages/PlannedNewOrderPage.tsx`:
- **Шаг 1** (pw1): категория (чип «изменить», как в прототипе — при повторном заходе можно сменить категорию, реализуется просто переходом на шаг 1) + описание + фото (переиспользует паттерн Фазы B: `POST /uploads` → `photoPaths`) + адрес+район (текстовые поля, без карты — плановый режим не использует геопозицию/`lat`/`lng` вообще, бэкенд-контракт `CreatePlannedOrderDto` их не принимает).
- **Шаг 2** (pw2): выбор даты (5 чипов — ближайшие 5 дней в пределах `PLANNED_HORIZON_DAYS=14`) + временной интервал (4 фиксированных слота: 08:00–10:00, 10:00–13:00, 13:00–16:00, 16:00–19:00 — комбинируются с выбранной датой в реальные `slotStart`/`slotEnd` ISO-строки на фронте) + бюджет (необязательное числовое поле).
- **Шаг 3** (pw3): предпросмотр «как увидят мастера» (без точного адреса/телефона — просто текстовая сводка из уже введённых данных, без доп. запроса) + кнопка «Опубликовать заявку» → `POST /planned-orders` с полным телом (`categoryId, description, address, district, entrance?, floor?, apartment?, addressComment?, slotStart, slotEnd, budget?, photoPaths?`).

### 2. `/planned/:id` — pwait: список ставок с рейтингом

Бэкенд уже обогащает каждую ставку (`enrichBids()`, Cycle 1): `rating`, `reviewCount`, `completedCount`, `experienceYears`, `verified`. Прототип показывает это на каждой карточке ставки (строки 511-529) — есть все данные, ничего выдумывать не нужно.

- Пустое состояние (0 ставок) — спиннер + текст «мастера уже видят заявку» (честно, не выдумка — заявка правда уже видна мастерам через `feed()`).
- Бейдж позиции на карточке — вычисляется на фронте: у ставки с минимальной ценой среди текущих — зелёный бейдж «Лучшая цена», у остальных — без бейджа (упрощение относительно прототипа, где показан только 1 демо-бейдж; не более одного правила ранжирования, чтобы не изобретать невалидируемую метрику «лучшее предложение»).
- «Выбрать» на карточке ставки → не сразу `select()`, а локальное состояние `pconfirm` (см. ниже).
- Кнопка «Сравнить (N)» → `/planned/:id/compare`.
- Кнопка «Отменить» → `POST /planned-orders/:id/cancel`.

### 3. `/planned/:id/compare` — таблица сравнения

Прототип жёстко показывает 2 мастеров бок о бок; реальных ставок может быть до 5 (`PLANNED_MAX_BIDS`). Таблица строится динамически на все текущие ставки (горизонтальный скролл при 3), колонки: аватар-инициалы+имя, строки: Цена/Рейтинг/Заказов/Опыт/Срок/Комментарий. Кнопка «Выбрать {имя}» под каждой колонкой → тот же `pconfirm`-переход, что и с карточки в pwait (общий колбэк, не дублируется).

### 4. pconfirm — подтверждение выбора (локальное состояние, не маршрут)

После тапа «Выбрать» (из pwait или compare) — рендерится сводка выбранной ставки + предупреждение «мастер получит точный адрес и телефон, должен подтвердить в течение 2 часов» (текст соответствует реальному `PLANNED_CONFIRM_TIMEOUT_S = 2*3600`) + кнопка «Подтвердить выбор», которая ТОЛЬКО ТЕПЕРЬ вызывает `POST /planned-orders/:id/select` с `{bidId}`. Кнопка «Назад к предложениям» просто сбрасывает локальное состояние.

### 5. pactive — два подсостояния одного вида

- **Ждём подтверждения** (`MASTER_SELECTED`): спиннер + «Ждём подтверждения от {имя} · обычно 10–20 минут · максимум 2 часа» (статичный текст, не точный таймер — прототип тоже не считает точно, а бэкенд не отдаёт отдельный `confirmDeadline` в формате, который стоило бы here превращать в обратный отсчёт; в отличие от Фазы B `priceDeadline`, здесь нет UI-сценария, где точный отсчёт критичен — упоминание «максимум 2 часа» и есть вся нужная информация пользователю).
- **Подтверждено** (`CONFIRMED`/`IN_PROGRESS`): зелёная плашка «{имя} подтвердил — контакты открыты» + карточка мастера (имя, `✓` — тот же безусловный инвариант «назначенный мастер = ACTIVE», что в Фазе B) + кнопка звонка (`tel:`, телефон уже не редактирован бэкендом на этой стадии) + сводка (цена из выбранной ставки = `workPrice`, «когда» из `slotStart`/`slotEnd`).
- Кнопка «Отменить заявку» — весь диапазон статусов до `DONE` включительно, как в старом коде.

### 6. done / closed — переиспользование паттерна Фазы B, не файлов

Отдельные `PlannedDoneView`/`PlannedClosedView` (не буквально импорт `DoneView`/`ClosedView` из Фазы B — разная форма цены: `workPrice` без `calloutPrice`/`serviceFee`-разбивки, т.к. у плановых заявок нет сбора за выезд). Рейтинг на `closed` — тот же принцип, что в Фазе B: `order.review` уже приходит из `ReviewsService.attachRating` (подключено в `planned-orders.service.ts` в этой же сессии, Rating-фича), звёзды → `POST /planned-orders/:id/review`.

### 7. Спор — рефакторинг, не дублирование

`DisputePage.tsx` (Фаза B) получает проп `kind: 'orders' | 'planned-orders'`, используемый для построения путей (`/${kind}/${id}/disputes`). Два маршрута (`/order/:id/dispute`, `/planned/:id/dispute`) рендерят один и тот же компонент с разным `kind`. `PATCH /disputes/:id` и `POST /disputes/:id/evidence` не зависят от типа заявки (общая `Dispute`-модель) — не меняются.

## Явно вне скоупа

- Мастерская сторона визарда (confirm/decline/on-site/complete через веб-UI) — нет фронтенд-страниц для роли мастера вообще, не только в плановом режиме; отдельный будущий цикл.
- Лид-кредиты — по контракту полностью невидимы клиенту (только сторона мастера), в клиентском UI не отображаются нигде.

## Самопроверка

Все 3 экрана визарда, pwait/compare/pconfirm/pactive, done/closed учтены. Контракты сверены напрямую с кодом (`planned-orders.controller.ts`, `.service.ts`, `dto.ts`, `planned-order.constants.ts`) — включая обнаружение реального бага в текущем `PlannedNewOrderPage.tsx`. Единственное решение, не буквально совпадающее с прототипом — бейдж позиции ставки упрощён до одного правила (дешевле — «лучшая цена»), явно зафиксировано, не оставлено как TBD.
