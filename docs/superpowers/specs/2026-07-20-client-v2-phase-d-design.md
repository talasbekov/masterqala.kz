# Клиент v2 — Фаза D (финальная): история, профиль, адреса, вспомогательные экраны — дизайн

> Завершающая фаза Цикла 2. Архитектурные решения (токены `c2`, i18n, `AppShell` для таб-бар-маршрутов vs старый `Layout` для drill-down) — те же, что в Фазах A/B/C.

## Скоуп

Экраны прототипа `orders, detail, profile, addresses, payments, support` (строки 623-833, 763-820 `Этап 5 - Клиент (mobile).dc.html`). `notifications` **уже реализован в Фазе A** (честный empty state) — Фаза D его не трогает.

## Маршруты

| Маршрут | Экран(ы) прототипа | Shell | Примечание |
|---|---|---|---|
| `/orders` | orders | `AppShell` (переезжает из старого `Layout` — теперь v2, как остальные таб-бар-маршруты) | вкладки «Активные»/«История» |
| `/profile` | profile | `AppShell` (переезжает) | + переключатель языка |
| `/profile/addresses` | addresses | старый `Layout` (drill-down, без таб-бара — как `/order/:id/dispute`) | полноценный CRUD |
| `/profile/payments` | payments | старый `Layout` | статичная заглушка «скоро» |
| `/support` | support | старый `Layout` | заменяет заглушку-редирект из Фазы A |

`detail` (просмотр закрытой заявки из истории) — **переиспользует уже готовые `/order/:id` и `/planned/:id`** (их `ClosedView`/`PlannedClosedView` из Фаз B/C), не отдельный экран с хронологией событий. Прототип показывает более богатую версию (лог событий по таймстемпам, разбивка цены, карточка мастера со звёздами) — сознательное упрощение: вся эта информация технически доступна (поля `acceptedAt`/`onSiteAt`/`priceProposedAt`/`completedAt`/`closedAt` есть в модели), но строить отдельный «receipt»-экран с таймлайном — заметное расширение скоупа этой уже большой фазы; уже готовый `ClosedView` показывает исход (выполнено/отменено) и рейтинг, этого достаточно для MVP истории. Фиксируется явно, не оставлено как TBD.

## Ключевые решения

### 1. MyOrdersPage v2 — вкладки «Активные»/«История»

`GET /orders` (`listMine`) + `GET /planned-orders/mine` — уже используются старым кодом, переиспользуются как есть. Активные — статусы вне терминальных (`!isTerminalStatus`/`!isPlannedTerminalStatus`, уже есть в `orderStatus.ts`). История — терминальные статусы (CLOSED/CANCELLED_*/EXPIRED/NO_MASTERS), отсортированы по дате. Клик по карточке → `/order/:id` или `/planned/:id` (уже построенные статус-driven страницы сами покажут нужный вид).

### 2. ProfilePage v2 — переключатель языка реально работает

Профиль — первый экран, где переключатель RU/KK/EN из хендоффа (строки 770-772) реализуется по-настоящему: `i18n.changeLanguage('kk'|'en'|'ru')` — реальный вызов `react-i18next`, не декорация. Т.к. `kk.json`/`en.json` — пустые заглушки с `fallbackLng: 'ru'` (решение Фазы A «только инфраструктура + RU-контент»), переключение физически меняет активный язык i18n (кнопка корректно подсвечивается активной), но видимый текст остаётся русским — честно, не выдаётся за законченный перевод.

Карточка «Стать мастером» — копия из прототипа содержит мета-фразу «Прототип мастера — отдельный файл этапа 5», обращённую к читателю дизайн-документа, не к конечному пользователю продукта — не переносится дословно, реальная подпись: «Зарабатывайте на своих навыках — верификация за 1–2 дня.» (первая половина фразы, без мета-ссылки на другой файл прототипа).

Ссылки на `/become-master`, `/wallet`, `/admin` (для OPERATOR) — сохраняются как есть, ведут на уже существующие нередизайненные (вне скоупа Цикла 2 — роль мастера/оператора) страницы. Тот же паттерн переходного состояния, что уже был на `/orders` и `/profile` все три предыдущие фазы.

### 3. AddressesPage — первый полноценный CRUD для `/addresses`

Бэкенд полностью готов с Cycle 1 (`GET/POST/PATCH/DELETE /addresses`), Фазы B/C только ЧИТАЛИ список для чипов быстрого выбора. Фаза D — первый и единственный экран с полным управлением: список + добавление + редактирование + удаление. Поля формы — те же, что уже использовались в мастер-данных: `label, address, entrance?, floor?, apartment?, comment?, isDefault`. `lat`/`lng` — не собираются (нет карты на этом экране, как и в самом прототипе) — оставляются `undefined`, бэкенд принимает без них.

### 4. PaymentsPage — честная заглушка, не выдуманные карты

Прототип показывает демо-карты «Kaspi Gold ···· 4821»/«Halyk ···· 0193» как будто реальные — решение пользователя («статичная заглушка «скоро»») явно отклоняется от этого: реальных карт нет, показывать несуществующие финансовые данные пользователя — грубое нарушение принципа «не выдумывать данные». Экран — иконка + текст «Управление способами оплаты скоро появится» + пояснение про существующий реальный флоу (резервирование на выезд/оплата после подтверждения — то же самое сообщение, что уже в прототипе снизу, оно фактически верно и не про несуществующие карты).

### 5. SupportPage — статичный FAQ (честно, без функциональности «ответа»)

4 вопроса из прототипа (дословно) — рендерятся как список заголовков без содержимого ответа (в прототипе у них нет реального перехода/раскрытия — не выдумывать несуществующий контент ответов). Кнопка «Позвонить: 7666» — `tel:`-ссылка, как и в прототипе.

## Явно вне скоупа

- Экран `detail` с логом событий по таймстемпам — сознательно не строится отдельно (см. §Маршруты выше).
- Раскрывающиеся ответы на FAQ — нет реального контента, не выдумывается.
- Полноценный переведённый KK/EN — только инфраструктура (решение Фазы A).

## Самопроверка

Все экраны прототипа (orders/detail/profile/addresses/payments/support) учтены — detail явно замаплен на уже готовые `/order/:id`/`/planned/:id` с зафиксированным обоснованием упрощения, notifications явно исключён (уже готов с Фазы A). Payments — явное, обоснованное отклонение от буквального прототипа (не выдуманные карты) по прямому решению пользователя.
